# [4.10] テクニカルライティング：ターゲット特定と情報の断捨離

「誰に届けるか」を決めない文章は、誰にも届きません。

## 1. 「誰が読むか」を決める (Audience Detection)
- **経営層**: コスト、期間、ビジネスリスクしか興味がありません。技術用語（シャフル、パーティション）は一切排除します。
- **エンジニア**: 実装方法、エラー時の挙動、最新技術に興味があります。
- **運用担当**: 毎日どこを見ればいいか、異常時の連絡先にしか興味がありません。

## 2. ワントピック・ワンメッセージ
一つの文（。まで）には、一つのことだけを書きます。
- **NG**: 「このジョブはメモリを16GB使っていますが、昨日は8GBで足りていたのに今日はエラーが出たので設定を変えた方が良いと思われます」
- **OK**: 「ジョブのメモリを倍増させる必要があります。理由は、今日の実行中に OOM（メモリ不足）が発生したためです。」

## 3. 「専門用語」の翻訳
「物理層でリパーティションしました」と顧客に言っていませんか？
「データの並び順を整理し直して、計算を効率化しました」と、**相手の頭にある言葉** に翻訳して伝えましょう。

## 🚨 The Incident: 全く読まれない 100ページの分厚いマニュアル
**事象**: 丁寧に 100ページ の Word マニュアルを作ったが、障害発生時に誰もそれを読まず、結局あなたに電話がかかってきた。
**原因**: 「どこに何が書いてあるか」がわからず、情報が多すぎたため。
**対策**: まずは **「1枚でわかるクイックスタート」** を作り、詳細はリンク先へ飛ばす、2段構えの構成にします。

## 🎯 Mastery Mission: 要約の訓練
**タスク**: あなたは「Databricks の Unity Catalog 導入によるセキュリティ強化」について、30分のプレゼン資料を作りました。
- 資料を読み返す時間がない忙しい役員に、エレベーターの中で **30秒（150文字程度）** で内容を報告してください。
- どのような「ベネフィット（利益）」を強調しますか？
**回答基準**: 技術的な詳細（GRANT文など）を捨て、ビジネス的な価値を伝えてください。
