# [4.18] ケーススタディ：大規模マイグレーションのプロジェクト完遂

旧世代（On-premse / Hadoop）から新世界（Databricks）への「引越し」をリードする。

## 1. 現場の状況
- **旧環境**: オンプレミス Hadoop (Hive)。メンテナンスコストが月 1,000万円。
- **課題**: 1,000 個もの SQL ジョブを、壊さずに Databricks へ移行しなければならない。

## 2. 移行戦略：3段階アプローチ
1.  **Lift & Shift**: まずはコード（SQL）を変えずに Databricks 上で動かす（互換性の確認）。
2.  **Refactor**: 重い処理を Delta Lake や Spark SQL の独自関数に書き換えて高速化。
3.  **Modernize**: DLT や Workflow を導入し、運用を全自動化。

## 3. データの「一致性」確認（Validation）
移行前後で 1円 でも数値がズレれば、プロジェクトは失敗です。
- **テクニック**: 旧環境の出力と新環境の出力を、Databricks 上で `EXCEPT` かけて「差分 0件」であることを全テーブルで自動検証するノートブックを作成。

---
## 🚨 The Incident: UDTF/UDF の泥沼
**事象**: 旧 Hive で使っていた Java 製の独自関数（UDF）が、Databricks 内で全く動かず、100個 の重要な集計が全滅した。
**対策**: マイグレーションの「前」に、**「特殊な関数の棚卸し（インベントリ調査）」** を行い、Spark で書き換え可能かを事前に検証しておく必要があります。

---
## 🎯 Mastery Mission: 移行の優先順位
**タスク**: 移行対象のテーブルが 1,000 あります。期間は 3ヶ月 です。
- 全てを一度に移行する「ビッグバン移行」と、重要なものから少しずつ移す「段階的移行」、どちらを選びますか？
- その際、移行が終わったものと終わっていないものが混在する「並行稼働期間」のデータ整合性をどう保ちますか？
