# [4.22] DoD (Definition of Done : 完了の定義) の合意形成

「よし、終わりました！」と報告した際に、相手（上司や顧客）に「え？テストは？設計書の更新は？」と返される……。このすれ違いを防ぐのが **DoD** です。

### 1. 「完了」のチェックリストを事前に作る
個別のタスクごとに「何をもって終わったことにするか」という条件を、関係者全員で合意しておきます。
*   例：[ ] コードがプッシュされている [ ] 単体テストが全てPassしている [ ] 設計書に最新の変更を反映している [ ] レビューを受けてApproveされている。

### 2. DoD を守るメリット
作業の質のバラツキがなくなります。さらに、「いつ終わるか」の予測が極めて正確になり、プロジェクトの炎上リスクが劇的に低下します。

---
**💡 実務のアドバイス:**
開発を始める「前」に、「今回の件の『完了の定義』を、こちらの5つのチェック項目で合意いただけますか？」と相手に投げてください。これだけで、成果物の手戻りはほぼゼロになります。
