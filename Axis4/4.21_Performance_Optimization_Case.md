# [4.21] ケーススタディ：性能ボトルネックの特定と解消法

実務で直面する「なぜか遅い」を、データと理論で切り裂く。

## 1. 現場の状況：売上集計ジョブの遅延
- **背景**: 毎晩 0時に 10TB の大規模集計を行っている。
- **事象**: 以前は 1時間 で終わっていたが、最近は 4時間 以上かかるようになり、朝のレポートが間に合わない。

## 2. アーキテクトの調査フロー
1.  **Spark UI の `Stages` タブ**: 
    - 200個のタスクのうち、1つだけが 1時間 走り続け、他は 1分 で終わっていることを発見（**Data Skew**）。
2.  **実行計画の `Explode`**: 
    - 特定のショップ ID にデータが極端に偏っていることが判明。
3.  **インフラ監視**: 
    - Worker の CPU 使用率は低いが、ネットワーク I/O が上限に張り付いている。

## 3. 解決した「打ち手」
- **コード**: AQE（Axis 2.08）の `skewJoin` を有効化。かつ、結合キーを「ID ＋ 年月」に複合化（Salting）。
- **インフラ**: SQL Warehouse のサイズを上げ、ボトルネックを解消。
- **結果**: 処理時間は **20分** に短縮された。

---
## 🚨 The Incident: 直感に頼った無駄な投資
**事象**: 「Worker が足りないせいだ」と判断し、台数を 128台 に増やしたが、速度は全く改善せず、コストだけが 10倍 に膨らんだ。
**教訓**: ボトルネックの正体（CPUか、メモリか、ネットワークか、コードか）を特定してから投資しましょう。
