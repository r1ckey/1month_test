# [4.16] ケーススタディ：性能ボトルネックの特定と解消

実弾演習。あなたは今、火が噴いているプロジェクトの救世主です。

## 1. 現場の状況
- **システム**: 日次売上集計ジョブ。
- **問題**: 半年前は 30分 だった処理が、今では 5時間 かかり、朝のレポートが間に合わない。
- **インフラ**: Standard Cluster (8台)。

## 2. 調査の 3ステップ
1.  **Spark UI の確認**:
    - `Spill (Disk)` が大量発生。特定のタスクだけが遅い（Data Skew）。
2.  **コードの精査**:
    - `groupBy().sum()` のキーが偏っている（Axis 2.07）。
    - かつ、保存前に無駄な `.cache()` が残っていた。
3.  **インフラの確認**:
    - VNet 内のトラフィックが Firewall を経由しており、帯域制限がかかっていた。

## 3. 解決した「打ち手」
- **コード**: AQE を有効化し、スキューのあるキーに Salting（Axis 2.07）を適用。
- **インフラ**: サービスエンドポイントを有効化し、Firewall をバイパス。
- **結果**: 処理時間は **15分** に短縮され、クラスタ費用も 1/4 に削減された。
