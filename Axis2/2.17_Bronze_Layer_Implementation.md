# [2.17] Bronze層の実装原則：一切の加工を捨て、履歴を残す

「オリジナルの証拠（エビデンス）」を守り抜く実装。

## 1. デザインパターン：インクリメンタル・ロード
Bronze層は、ソースシステムから「増分（新しく追加された分）」だけを読み込み、末尾に追記（`APPEND`）するのが基本です。
- **スキーマ**: ソースのスキーマが途中で変わっても、全ての列を保持し続けます。
- **メタデータ**: 読み込んだ日時（`_ingested_at`）や、元のファイルパス（`_source_file`）を付与するのが「実務の嗜み」です。

## 2. 禁忌事項
- **値の修正**: データが間違っていても、Bronzeでは直してはいけません。「間違ったレコードが届いた」という事実をそのまま記録します。
- **WHEREによる絞り込み**: 後の工程で「実はあの列も必要だった」となった時に困るため、全ての列を保存します。

## 3. コード例 (PySpark)
```python
df_raw = spark.read.format("csv").load(source_path)

# メタデータを付与して保存
from pyspark.sql.functions import current_timestamp, input_file_name

df_bronze = df_raw.withColumn("_ingested_at", current_timestamp()) \
                  .withColumn("_source_file", input_file_name())

df_bronze.write.format("delta").mode("append").saveAsTable("bronze.raw_sales")
```

---
## 🚨 The Incident: 型変換して保存したらエラー
**事象**: Bronze層で数値列を `int` に変換して保存しようとしたが、ソースに「文字列」が混ざっていて読み込みが止まった。
**解決策**: Bronzeでは型変換をせず、全て `STRING` で読み込んで保存します。型変換の格闘は Silver 層で行うのが正解です。
