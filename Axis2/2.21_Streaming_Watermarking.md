# [2.21] Watermarking（ウォーターマーキング）：遅延データの破棄とメモリ管理

「ストリーミングで、いつまでも来ないデータを待ち続ける」のをやめるための境界線。

## 1. 遅延データの課題
リアルタイム処理において、ネットワークの不調などで 1時間前のデータが今届くことがあります。
- **問題**: Sparkは「さらに古いものが来るかもしれない」と思い、計算結果をメモリに保持し続けます。
- **結果**: 保持し続ける期間が長すぎると、メモリ（State）が溢れてシステムが停止します。

## 2. Watermarking とは
「この時刻より前のデータが来ても、もう無視する」という足切りラインを定義することです。
```python
# 「10分以上前のデータは無視する」という設定
df_streaming = df.withWatermark("event_time", "10 minutes") \
                 .groupBy("event_time", "shop_id") \
                 .count()
```

## 3. 実務での「遅延許容範囲」の決め方
- **厳しすぎ（例: 1分）**: 少しのネットワーク遅延で、正当なデータが捨てられてしまい、売上集計が不正確になります。
- **ゆるすぎ（例: 24時間）**: 24時間分の全集計結果をメモリに持ち続けるため、バカ高いメモリが必要になります。

---
## 🚨 The Incident: 捨てられたはずのデータが残る？
**事象**: `withWatermark` を設定したが、古いデータがまだ集計に含まれている。
**解説**: Watermarking は「即座にファイルを消す」のではなく、**「次のマイクロバッチが走るタイミングで計算結果（State）から除外する」** 動きをします。
**教訓**: 動作確認の際は、数回マイクロバッチを走らせてから、古いデータが更新されなくなることを確認してください。
