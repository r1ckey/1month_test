# [2.13] OPTIMIZEコマンドの仕組みとファイル圧縮（Small File Problem対応）

Sparkは並列処理の特性上、何千もの「数KB」という極小ファイルを撒き散らすことがあります。これが **Small File Problem (小さなファイル問題)** です。

### 1. なぜ小さなファイルが悪いのか？
ファイルを開く・閉じるというオーバーヘッド（手続き）に時間が取られ、読み取りパフォーマンスが劇的に低下します。

### 2. OPTIMIZE による解決
`OPTIMIZE` コマンドを実行すると、Sparkはこれらの無数の小ファイルを読み込み、**「ちょうどいいサイズ（約1GB）」**の大きなファイルへ再構築（圧縮）して書き直します。
```sql
OPTIMIZE sales;
```

### 3. 自動最適化 (Auto Optimize)
Databricksには、書き込み時にリアルタイムでファイルをいい感じのサイズにする設定（`delta.autoOptimize.optimizeWrite`）もありますが、基本的には定期的なバッチとしての `OPTIMIZE` 実行との併用が一般的です。

---
**💡 チューニングの勘所:**
ストリーミング処理（1分ごとに少しずつデータが来る）テーブルは、放っておくと毎日数千のファイルが増えていきます。週次または日次の運用バッチで `OPTIMIZE` を組み込むのがアーキテクトの仕事です。
