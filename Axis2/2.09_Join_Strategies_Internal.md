# [2.09] Broadcast Join vs Sort Merge Join：実務での使い分け

「ネットワーク」を走らせるか、それとも「自分のメモリ」にコピーを置くか。

## 1. Broadcast Join (最強の速度)
- **仕組み**: 片方の小さいテーブルを全ての Worker のメモリに丸ごとコピー（配布）します。
- **メリット**: **シャフル（Shuffle）が発生しない**。100% 狭い変換（Narrow）として動くため、爆速です。
- **実務**: マスタデータ（都道府県名など）と、巨大なトランザクションデータを結合する際に必須です。
- **制限**: デフォルトでは 10MB 以下のテーブルにしか適用されません。

## 2. Sort Merge Join (堅牢な標準)
- **仕組み**: 両方のテーブルをキーでソート（並び替え）し、ノード間でシャフルして突き合わせます。
- **メリット**: **どんなに巨大なデータ同士でも確実に動作** します。メモリに入り切らなくてもディスクを使って完遂します。
- **デメリット**: 非常に重厚長大。シャフルのコストが大きいです。

## 3. 明示的な Broadcast の指示
```python
from pyspark.sql.functions import broadcast

# Sparkに「このテーブルは小さいから全てのWorkerに配れ（Broadcastしろ）」と命令する
df_final = df_large.join(broadcast(df_small), "id")
```

---
## 🚨 The Incident: Broadcast OOM
**事象**: `broadcast()` を使って高速化を狙ったが、実はそのテーブルが 2GB もあり、全ての Worker がメモリ不足で一気に全滅した。
**教訓**: 「小さい」の定義は、Workerのメモリ（Executor Memory）の空き容量に依存します。1GBを超えるようなテーブルを Broadcast するのは非常に危険です。
