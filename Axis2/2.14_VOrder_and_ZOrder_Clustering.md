# [2.14] V-Order と Z-Order：多次元クラスタリングによる爆速検索

「インデックス」が無いデータレイクで、どうやって目的の1行をマッハで探すか。

## 1. Z-Order クラスタリング
従来の「パーティション（フォルダ分け）」は1つの列（例：日付）でしか効きません。
Z-Orderは、**複数の列（例：日付 ＋ 会員ID）** を組み合わせて、データの配置を多次元的に最適化します。
- **効果**: 「特定の日付」かつ「特定の会員」のデータを検索したとき、無関係なファイルを 99% スキップ（Data Skipping）して、必要な数ファイルだけをピンポイントで読み込みます。
```sql
OPTIMIZE sales ZORDER BY (member_id, shop_code);
```

## 2. V-Order (Databricks 独自)
Parquet ファイル内部の「データの並び順」を、Databricks の計算エンジンが最も読みやすい形式に並び替えます。
- **実務**: Microsoft Fabric 等の他システムからも爆速で読み込めるようになるため、オープン性の観点でも推奨されます。

## 3. 実務での Z-Order 選定基準
- **注意**: 5つも6つも列を指定してはいけません。効果が薄れます。
- **定石**: 高頻度で `WHERE` 句や `JOIN` のキーに使われ、かつ種類が多い（カーディナリティが高い）列を 1〜3個 選ぶのが正解です。

---
## 🚨 The Incident: 全く効かない Z-Order
**事象**: `ZORDER BY (price)` （価格）を設定したが、検索が全く速くならない。
**原因**: 金額のような「ほぼバラバラ」で、範囲指定が多い列に Z-Order をかけても、ファイルの絞り込みが上手く機能しません。
**教訓**: IDやコード、カテゴリ名など、等価比較（`=`）によく使う列に絞りましょう。
