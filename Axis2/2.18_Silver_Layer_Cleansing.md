# [2.18] Silver層の実装論理：クレンジングと重複排除の戦い

「誰もが安心して信じられるデータ」への昇華。

## 1. Silver層で行うべき 4つのコト
1.  **データ型の強制**: 文字列だった `100` を `INT` に、`20240101` を `DATE` へ変換。
2.  **重複排除 (Deduplication)**: システムエラー等で二重に読み込まれたレコードを、同一IDで最新のものだけに絞り込む。
3.  **NULL処理・不要列の削除**: 解析に関係ないゴミデータを除去。
4.  **外部キーの整合性チェック**: 存在しないマスタIDを弾く、など。

## 2. 重複排除のコード例 (Window 関数)
実務で最も使われる「最新の1件だけを残す」ロジックです。
```python
from pyspark.sql.window import Window
from pyspark.sql.functions import row_number, col

# 同じ ID の中で、最新の更新時刻(updated_at)の行を 1位 とする
window_spec = Window.partitionBy("id").orderBy(col("updated_at").desc())

# 1位の行だけを抽出する
df_silver = df_bronze.withColumn("row_num", row_number().over(window_spec)) \
                     .filter(col("row_num") == 1) \
                     .drop("row_num")
```

## 3. 実務のコツ：UPSERT (MERGE INTO)
Delta Lake の `MERGE` 文を使い、既存データがあれば更新、なければ挿入する `UPSERT` を行います。これにより、Silver層は常に「最新の正しい状態」を保つことができます。

## 🚨 The Incident: タイムゾーンの混乱
**事象**: システム A (JST) とシステム B (UTC) のデータを Silver 層で結合したが、時間の計算が合わずに売上の集計が 9時間 ズレてしまった。
**教訓**: Silver層の最初のステップで、**全ての時刻データを UTC（世界標準時）に統一** する運用を徹底しましょう。表示する時だけ各国の時刻に直すのがアーキテクチャの鉄則です。

## 🎯 Mastery Mission: クレンジングの設計
**タスク**: あなたは「会員登録データ」を Silver 化します。
- `email` カラムに、明らかに不正な値（`test@test`, `aaaaa` など）が混じっています。
- あなたなら、これらをどのように処理しますか？
    1. プログラムでエラーを出して止める。
    2. 無視してそのまま Silver に入れる。
    3. 特殊なフラグ（`invalid_flag = True`）を立てるか、`NULL` で上書きする。
**回答基準**: 理由と共に「実務として最も運用が回る方」を選んでください。
