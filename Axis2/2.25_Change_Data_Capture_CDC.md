# [2.25] 変更データキャプチャ (CDC) の実装：APPLY CHANGES INTO の魔力

「削除された行」「更新された値」を、データレイク側でどう同期させるか。

## 1. CDC (Change Data Capture) とは
外部データベース（SQL Server, Oracle 等）で起きた「追加・更新・削除」の履歴（ログ）を読み取り、Delta テーブルを最新状態に同期させる手法です。
- 従来のやり方: `MERGE INTO` 文を自分で細かく書く必要があり、非常に複雑でした。

## 2. DLT の `APPLY CHANGES INTO`
DLT 独自の機能で、**「CDC ログを見て、最終的なテーブルを自動構築してくれ」** と一行指示するだけで完結します。
- **順序保証**: `SEQUENCE BY` 句により、同じ行に対する「更新 -> 削除 -> 更新」というイベントが前後しても、最終的に正しい状態になるよう制御します。

## 3. 実装イメージ
```python
dlt.apply_changes(
  target = "users_master",
  source = "cdc_raw_logs",
  keys = ["userId"],
  sequence_by = col("timestamp"),
  apply_as_deletes = expr("operation = 'DELETE'"),
  stored_as_scd_type = 1  # 1:上書き, 2:履歴保持
)
```

---
## 🚨 The Incident: 削除フラグの罠
**事象**: ソース側で論理削除（`is_deleted = true`）されたデータが、Delta 側では「行が残ったまま」になっていた。
**原因**: `apply_as_deletes` に論理削除のフラグを指定し忘れていた。
**教訓**: CDCには「物理削除（行が消える）」と「論理削除（フラグが立つ）」の2パターンがあります。どちらを「同期先のテーブルから消す」条件にするか、設計書での定義が必須です。
