# [2.08] Adaptive Query Execution (AQE)：Sparkに「任せる」最適化

「実行してみないとわからない」統計情報を、Sparkが走りながら判断します。

## 1. AQE とは何か
Spark 3.0 以降の目玉機能。実行計画を「動的」に書き換えます。
- **従来**: 実行前に「こう動く」と決めたら、途中でデータ量が予想と違っても変えられなかった。
- **AQE**: 最初のStageが終わった時のデータ量を見て、「次のStageのパーティション数を減らそう」とか「JOINの方式を Broadcast に変えよう」と自動判断します。

## 2. AQE の3大メリット
1.  **Coalescing Post-shuffle Partitions**: 200個に分けたけど中身が空っぽだったパーティションを、賢く 1つ にまとめてくれます。
2.  **Converting Sort-merge Join to Broadcast**: 統計情報が古くて「大きい」と思っていたテーブルが実は「小さかった」場合、爆速な Broadcast Join に切り替えます。
3.  **Skew Join Optimization**: スキュー（偏り）を自動検知して、タスクを分割してくれます。

## 3. 設定をオンにする
Databricks Runtime 7.3+ ではデフォルトでオンですが、念のため確認すべき設定です。
`set spark.sql.adaptive.enabled = true`

---
## 🚨 The Incident: 意図しないプラン変更
**事象**: 検証環境では上手くいっていたクエリが、本番の巨大データで AQE によって「予期せぬ Broadcast Join」に化け、メモリ不足（Broadcast OOM）を起こした。
**対策**: AQEは万能ではありません。Broadcast される最大サイズ（`autoBroadcastJoinThreshold`）を適切に（例：10MB〜100MB程度）制限しておくことが、本番運用の安全策となります。
