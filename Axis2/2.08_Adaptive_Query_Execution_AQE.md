# [2.08] Adaptive Query Execution (AQE) のダイナミック最適化

これまでのSparkは「実行前に立てたプラン」を頑固に守って走る仕組みでした。これを**「実行中の実測値」を元に柔軟に変更**するように進化したのが **AQE** です。

### 1. AQEの3つの魔法

1.  **Coalescing Post-shuffle Partitions**:
    シャッフル後、実はデータがスカスカだった場合、200個あったパーティションを自動で「8個」などに結合して効率化します。
2.  **Switching Join Strategies**:
    「当初はデカいと思ってたけど、フィルタした後ならこのテーブル、メモリに乗る（小さい）じゃん！」と判断し、途中で重たい Sort Merge Join から 爆速な **Broadcast Join** へ乗り換えます。
3.  **Optimizing Skew Join**:
    実行中に「あ、このキー偏ってるな」と気づいたら、自動的にパーティションをさらに細かく分割して並列度を上げます。

### 2. 有効化の設定
Databricks (Spark 3.x以降) ではデフォルトでONになっていますが、以下のフラグで制御されます。
`spark.sql.adaptive.enabled = true`

---
**💡 アーキテクトの視点:**
AQEの登場により、エンジニアが細かくパーティション数を指定しなくても「そこそこ速い」処理が手に入りました。しかし、AQEでも救えないレベルの超巨大データ（PB級）では、依然として手動の設計力が試されます。
