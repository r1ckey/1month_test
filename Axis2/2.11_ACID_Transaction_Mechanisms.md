# [2.11] ADLS上の ACIDトランザクションの仕組み：並列実行の競合解決

なぜ「複数のワークスペースから同時に書き込んでも壊れない」のか。その舞台裏の排他制御。

## 1. 楽観的同時実行制御 (Optimistic Concurrency Control)
Delta Lake は、書き込みを開始する際、他の誰も書き込んでいないと **「楽観的に（信じて）」** 処理を始めます。
- **書き込み完了時**: もし同時に誰かが `_delta_log` を書き換えていたら、Sparkは「競合が発生した」と判断し、自分の処理を一度捨ててマージし直すか、リトライを試みます。

## 2. 相互排他 (Mutual Exclusion) の実装
クラウドストレージ（ADLS等）には通常、ファイルをロックする仕組みがありません。
- **Databricks の解決策**: Azure Storage の API を使い、「ファイルが存在しない場合のみ作成する（Put if Not-Exists）」というアトミックな操作を利用して、000001.json 等の連番ログを作成することで、擬似的にロックを実現しています。

## 3. 実務での「上書き (Overwrite)」の安全性
従来のParquetでは、上書き中に誰かが読み取ると「不完全なデータ」を読むリスクがありました。
- **Delta**: 新しいファイル群を書き終え、最後にログに「今からこっちが正解！」と一行書き加えるまで、古いデータが有効であり続けます。読み手は常に「100%正しいスナップショット」を見ることが保証されます。

---
## 🚨 The Incident: コンフリクトエラー
**事象**: 同じテーブルに対して、バッチJob Aが `INSERT` し、バッチJob Bが `UPDATE` していたところ、Job Bが `ConcurrentAppendException` で落ちた。
**原因**: 同じ行やパーティションを同時に編集しようとしたため。
**対策**: ジョブの実行タイミングをずらすか、書き込み対象のパーティションを明確に分ける（Parititioning）ことで競合を回避します。
