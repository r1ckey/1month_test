# [2.02] RDD, DataFrame, Dataset API の進化と Catalyst オプティマイザ

「どう書くのが一番速いか」という古くて新しい問題。

## 1. APIの歴史的変遷
1.  **RDD (Resilient Distributed Dataset)**:
    - 超低レイヤー。自由度は高いが、Sparkが「何をしているか」を理解できないため、最適化が全く効かない。
    - **現代の実務**: ほぼ使いません。
2.  **DataFrame / SQL**:
    - テーブル形式の構造化データ。Sparkが「どの列が必要か」を理解できる。
    - **Catalyst**: Sparkの超高性能な脳みそ。SQL文を解析し、最も効率的な実行順序に勝手に組み替えてくれます。
3.  **Dataset (Scala/Java)**:
    - 型安全。ただし、Pythonには関係ありません（PythonはDataFrame一択）。

## 2. Catalyst オプティマイザの凄さ
あなたが「全ての列を読み込んでからフィルタする」という下手なコードを書いても、Catalystは **「先にフィルタしてから、必要な列だけ読み込む」** という実行計画（Predicate Pushdown / Projection Pushdown）に勝手に修正してくれます。

## 3. Python (PySpark) での注意
PySparkは内部的に「Pythonプロセス」と「JVMプロセス（Spark本体）」が通信しています。
- **UDF (User Defined Function)** の乱用は、このプロセス間通信のオーバーヘッドにより、速度を 10倍以上 低下させることがあります。
- **原則**: 可能な限り PySpark の組み込み関数 (`pyspark.sql.functions`) を使い、自作関数（UDF）は最後の手段にしましょう。

---
## 🚨 The Incident: 「とりあえずUDF」で大炎上
**事象**: 日付フォーマットの変換を Python の `datetime` モジュールを使った UDF で実装したところ、1000万件の処理に1時間かかった。
**解決策**: PySpark 標準の `to_date` 関数に書き換えたところ、**3分** で完了した。
**教訓**: Spark組み込み関数は「C++やJavaの爆速レイヤー」で動きます。Pythonで書くのはできるだけ避けましょう。
