# [2.05] メモリ OOM (Out Of Memory) の根本原因とメモリ管理の鉄則

「とりあえずデカいマシン（Worker）にすれば治る」は迷信です。

## 1. Spark のメモリ構造
Workerのメモリは、主に2つの領域に分かれています。
1.  **Execution Memory**: 計算用（JOINや集計作業用）。ここがいっぱいになると、データがディスクに溢れ（Spill）、激重になります。
2.  **Storage Memory**: キャッシュ用（`df.cache()` で保持される部分）。

## 2. 3大 OOM 原因
1.  **Driver OOM**: `collect()` で大量のデータを一箇所に集めすぎた。（前章で説明）
2.  **Executor OOM**: 一つのタスクが大きすぎて（Data Skew）、1台の Worker の限界を超えた。
3.  **Broadcast OOM**: 小さいテーブルを全Workerに配る（Broadcast）つもりが、実はおまけのテーブルが大きすぎてメモリを食いつぶした。

## 3. 実務での「キャッシュ（Cache）」の呪い
「速くなるから」と、あらゆるところで `.cache()` や `.persist()` を書くのは **アンチパターン** です。
- ストレージメモリを占領し、逆に Executor の計算用メモリを圧迫します。
- さらに、キャッシュを使い終わった後に `.unpersist()` し忘れると、クラスタが「メモリ不足」で窒息します。

---
## 🚨 The Incident: 全く終わらない `Spill to Disk`
**事象**: Spark UI で `Spill (Memory) 50GB / Spill (Disk) 10GB` という恐ろしい表示が出ている。
**原因**: Workerのメモリに入り切らないデータが、仕方なくハードディスクに書き出されて処理されている。速度は 100倍 遅くなります。
**解決策**: Workerの台数を増やすか、フィルタをかけて処理対象件数を減らす必要があります。
