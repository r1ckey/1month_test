# [2.05] OOM (Out of Memory) の根本原因と Executor メモリ管理限界

「`java.lang.OutOfMemoryError`」
データエンジニアを深夜に叩き起こす、最悪のメッセージです。

### 1. Driver OOM (司令塔が死ぬ)
*   **原因**: `collect()` で分散されていた全データを1箇所に集めすぎた。またはNotebookの出力結果が大きすぎた。
*   **対策**: `take()` で小さなサンプルだけ取得するか、直接ADLSへ `write()` し、Driverには何も返さない。

### 2. Executor OOM (現場の作業員が死ぬ)
*   **原因**: 1つのコア（Task）で扱おうとしているデータ量が、そのWorker VMの「メモリ」を溢れてしまった。
*   **原因詳細**: 重たい関数（Python内蔵処理や、不適切なJOIN）で、一時的な計算バッファが激増した。

### 🌟 メモリ構成の理解
Sparkは1つのJVM（Java VM）の中で、「計算用(Execution)」と「キャッシュ用(Storage)」にメモリを分けて使っています。

---
**💡 トラブルシューティング:**
頻繁にOOMが発生する場合、安易に「デカいマシンに変える」前に、**パーティション数（並列度）を増やして、1つのTaskが扱う負担を小分けにする**ことで解決することが多々あります。
