# [2.23] Delta Live Tables (DLT) 入門：宣言型データエンジニアリング

「どうやって作るか（How）」ではなく、「何を作るか（What）」を記述する次世代のパイプライン。

## 1. DLT とは
従来の Spark Job は「データ A を開く -> 加工する -> B に書き込む」という手順を人間が細かく指示していました。
DLT は **「テーブル B は テーブル A を元にこのように定義される」** という依存関係を宣言するだけで、Databricks が裏側で最適な実行順序、チェックポイント、リトライを全て管理します。

## 2. DLT のメリット
- **自動 DAG 管理**: テーブル間の親子関係を GUI で美しく可視化。
- **Auto-Scaling**: データ量に応じて、必要なクラスタ数を最小〜最大の間で自動調整。
- **テストの統合**: 次の 2.24 で学ぶ「品質チェック（Expectations）」を標準装備。

## 3. 基本的な書き方 (Python)
```python
import dlt

@dlt.table
def click_logs_bronze():
  return spark.readStream.format("cloudFiles").load("/data/raw")

@dlt.table
def click_logs_silver():
  return dlt.read("click_logs_bronze").filter("user_id IS NOT NULL")
```

## 🚨 The Incident: 複雑なデバッグが困難
**事象**: DLT パイプラインが途中でエラーになったが、通常の Spark ログと見方が異なり、どこでデータが壊れたのかの切り分けに苦労した。
**対策**: DLT には専用の **「イベントログ」** があります。また、開発中は「Continuous（継続実行）」ではなく「Manual（手動実行）」モードを使い、エラー箇所を一段階ずつ特定するフローを身につけましょう。
