# [1.14] Azure Key Vault 連携と Secrets (シークレット) の運用設計

「パスワードをソースコードに書く（ハードコード）」は、エンジニアとしての死を意味します。

## 1. Secrets (シークレット) 管理とは？
DBのパスワード、APIトークン、ストレージのアクセスキーなどを、暗号化された安全な金庫に保管し、コードからは別名（エイリアス）で呼び出す仕組みです。

## 2. Azure Key Vault (AKV) バックエンドの構築
Databricks独自の金庫（Databricks-backed）も作れますが、実務では **「Azure Key Vault」** をバックエンドにするのが黄金律です。
- **統合管理**: Azure portal側で一括してパスワードを変更すれば、Databricks側も自動で最新になります。
- **設定手順**:
    1. Azureポータルで Key Vault を作る。
    2. `https://<adb-url>#secrets/createScope` を開き、Key Vaultの名前とDNS名を登録する。

## 3. Pythonコードでの呼び出し
```python
# シークレットの取得（画面上に生の値は表示されない！）
db_password = dbutils.secrets.get(scope="my-key-vault", key="sql-pass")

# これを使ってDB接続
# url = f"jdbc:sqlserver://...;password={db_password}"
```

---
## 🚨 The Incident: `print()` しても中身が見えない！
**事象**: デバッグのために `print(db_password)` を実行したが、出力が `[REDACTED]` と表示されて値が確認できない。
**解説**: これは **Databricksの高度なセキュリティ機能** です。ログにパスワードが漏れるのを防ぐため、シークレットの中身は強制的に隠蔽されます。
**教訓**: 「見えないのは不具合ではない」と理解し、値の正しさは接続テストの成功有無で判断しましょう。
