# [1.19] Databricks Asset Bundles (DABs) 概念：プロジェクト構成の標準化

「単なるNotebookの寄せ集め」から、「一つのソフトウェア開発プロジェクト」へ。

## 1. DABs とは何か？
2024年現在のDatabricks開発における **デファクトスタンダード** です。
- **YAMLによる定義**: Notebook、Job、Pipeline（DLT）、設定を一つのパッケージ（Bundle）として管理します。
- **ビルドとデプロイ**: `databricks bundle deploy` コマンドだけで、Git上のソースコードを各環境（dev/stg/prod）へ安全にデプロイできます。

## 2. なぜDABsが必要なのか
これまでは、NotebookのデプロイはGit連携で行い、Jobの設定は画面で手動……という「つぎはぎ」の運用でした。
DABsを使えば、**「このJobはこのNotebookを呼び、この設定値（環境変数）を使う」** という全ての関係性を一つのプロジェクトとして一貫性を持って管理できます。

## 3. プロジェクト構造例
```text
my-project/
├── databricks.yml    # 全体設定
├── bundle.yml        # リソース定義
├── src/              # Notebook や Python ファイル
│   ├── bronze_ingest.py
│   └── silver_transform.py
└── resources/        # Jobの定義など
    └── my_job.yml
```

## 🚨 The Incident: 複数人での開発が混乱
**事象**: AさんとBさんが同じ開発ワークスペースにデプロイしたところ、お互いのJob設定が上書きされてしまった。
**解決策**: DABsの設定ファイルで `development` モードを使い、デプロイされるJobの名前に自動的にユーザー名（例： `job_john_doe`）が含まれるように設計し、リソースの競合を避けます。
