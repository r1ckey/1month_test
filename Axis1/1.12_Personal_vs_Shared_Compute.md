# [1.12] パーソナル・コンピュート vs 共有クラスタ：同時実行性と分離

「一つのクラスタをみんなで使うか、一人一台持つか」
コストと使い勝手のトレードオフを決定する重要な選択。

## 1. 共有クラスタ (Shared Access Mode)
- **特徴**: Unity Catalogが有効な環境での標準。複数のユーザーが同じクラスタ上で計算を行う。
- **メリット**: リソースを効率的に使い回せる。UCの権限管理（行・列フィルタ）が効く。
- **デメリット**: ある人が重いクエリを投げると、他の人のNotebookも重くなる（リソース競合）。

## 2. シングルユーザ (Single User Access Mode)
- **特徴**: そのクラスタには「一人」しかアタッチできない。
- **メリット**: その人の専用リソース。他人に邪魔されない。Python/Rのライブラリ競合も起きない。
- **用途**: 複雑な機械学習モデルの学習や、高度なライブラリを多用する開発。

## 3. サーバレス (Serverless) SQL / Compute
- **特徴**: クラスタの起動待ち時間（通常3〜5分）がゼロ。
- **メリット**: 管理画面を触る必要すらない。使った秒数分だけ課金。
- **実務判断**: ダッシュボードの閲覧や、数分の分析であればサーバレスが圧倒的にコスパが良いです。

---
## 🚨 The Incident: ライブラリの衝突
**事象**: 共有クラスタ上で、Aさんが `pandas 1.0` を入れ、Bさんが `pandas 2.0` を上書きで入れようとし、環境が壊れた。
**解決策**: 共有クラスタではクラスタレベルのライブラリ変更を制限し、Notebookローカルなインストール (`%pip install`) を推奨するか、そもそも個人での開発は「シングルユーザ」クラスタに分離します。
