# [1.18] Terraform (IaC) を用いた Databricks リソースの自動構築入門

インフラをコードで管理する **「IaC (Infrastructure as Code)」** の代表格が Terraform です。Databricks専用の「Databricks Provider」が存在します。

## 1. なぜ Terraform を使うのか？
*   **再現性**: 開発(Dev)環境で作ったのと「全く同じ設定」のクラスタや権限を、本番(Prod)環境に一瞬で複製できます。
*   **履歴管理**: 設定の変更履歴が Git（GitHub等）に残るため、誰がいつ設定を変えたか一目瞭然です。

## 2. 基本的な HCL (HashiCorp Configuration Language) の例
```hcl
# クラスタの作成定義
resource "databricks_cluster" "shared_cluster" {
  cluster_name            = "Project_Shared_Cluster"
  spark_version           = "13.3.x-scala2.12"
  node_type_id            = "Standard_DS3_v2"
  autotermination_minutes = 20
  
  autoscale {
    min_workers = 1
    max_workers = 3
  }
}
```

## 3. Azure Provider との組み合わせ
Databricksワークスペース本体は `azurerm` プロバイダーで作り、その中身のリソース（クラスタ等）は `databricks` プロバイダーで作るという二段階構成になります。

> [!IMPORTANT]
> **State ファイルの管理**
> Terraformの実行状態を記録する `.tfstate` ファイルは、絶対に個人のPCに置いてはいけません。Azure Storage Account (Blob) 上に「リモートバックエンド」として保管し、チームで共有するのが鉄則です。

---
**💡 実務のヒント:**
大規模プロジェクトでは「手作業のクラスタ作成」は原則禁止とし、全てPull RequestベースのTerraform適用で管理するスタイルが主流です。
