# [1.18] Infrastructure as Code (Terraform) を用いた自動構築

「誰が作っても同じインフラ」を保証し、手作業によるヒューマンエラーを殲滅します。

## 1. Terraform for Databricks
Databricksは公式の **「Databricks Provider」** を提供しており、Azureのリソース（ワークスペース自体）と、Databricks内部のリソース（クラスタやUC）の両方を管理できます。

## 2. 実務でのメリット
- **環境複製 (Replication)**: 「開発環境を完璧にコピーして、30分で検証環境を作る」ことが可能です。
- **変更履歴の管理**: 「誰がクラスタの最大台数を変えたのか」をGitのコミット履歴で追跡できます。
- **削除忘れの防止**: `terraform destroy` 一発で、不要になった検証環境を綺麗に掃除できます。

## 3. 基本的なコード構成
```hcl
# クラスタの定義例
resource "databricks_cluster" "shared_autoscaling" {
  cluster_name            = "Shared-Autoscaling-Cluster"
  spark_version           = "14.3.x-scala2.12"
  node_type_id            = "Standard_DS3_v2"
  autotermination_minutes = 20
  autoscale {
    min_workers = 1
    max_workers = 10
  }
}
```

## 🚨 The Incident: Terraform Drift (乖離)
**事象**: 誰かがポータルから勝手にクラスタ名を書き換えたため、Terraformを実行すると「上書きして元に戻そう」としてしまい、予期せぬ再起動が発生した。
**解決策**: 本番環境では **「ポータルからの編集禁止（ReadOnly）」** を徹底し、全ての変更は必ずTerraform（Pull Request）を通じて行う運用ルールを確立します。
