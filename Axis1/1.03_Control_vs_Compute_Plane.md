# [1.03] Control Plane と Compute Plane の物理的境界と責任分界点

「データはどこにあるのか？」これを顧客に説明できることが、アーキテクトに求められる説明責任（Accountability）です。

## 1. 二つの平面（Planes）
- **Control Plane**:
    - **場所**: Microsoft/Databricksが管理するサブスクリプション。
    - **役割**: UIの提供、Jobのスケジューリング、クラスタの管理。
    - **保持データ**: Notebookのソースコード（メタデータ）、ユーザー情報。**実際のデータ（CSVやテーブル）はここには入りません。**
- **Compute Plane**:
    - **場所**: **あなたの** Azureサブスクリプション。
    - **役割**: データの計算（Spark）、外部ストレージへの書き込み。
    - **保持データ**: Workerノード内部のテンポラリファイル。

## 2. セキュリティ・コンプライアンス上の意義
「データはあなたの自社環境（Compute Plane）から一歩も外に出ません」という魔法の言葉（文言）は、法務やセキュリティチェックをパスする際の切り札になります。

## 3. 責任分界点 (Shared Responsibility Model)
- **インフラのパッチ適用**: DatabricksがOSイメージ（DBR）を更新。あなたは「DBRバージョンを上げる」だけでOK。
- **Notebookの暗号化**: Control Plane内のコードは、デフォルトでAES-256暗号化されますが、「自分で用意した鍵（Customer-Managed Key : CMK）」で暗号化することも可能です（上級要件）。

---
## 🚨 The Incident: 機密データがControl Planeに漏れる？
**事象**: Notebookの中に `password = "xxxxx"` と直書きしてしまった。
**リスク**: これは Control Plane（DBX管理下）に永続保存されます。
**対策**: 必ず **Azure Key Vault** や **Secrets API** を使い、コードには変数名だけを書くように設計書で規定します。
