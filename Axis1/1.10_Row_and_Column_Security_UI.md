# [1.10] 行・列レベルのセキュリティ（Row & Column Security）

「同じテーブルを見ているのに、見える範囲が人によって違う」という魔法の実装。

## 1. なぜこれが必要か？
- **動的マスキング（列レベル）**: 一般社員には「メールアドレス」を `***@***` と隠して見せ、人事担当にはそのまま見せる。
- **行フィルタリング（行レベル）**: 東京支店の人は「東京の売上データ」しか見えず、全社マネージャーは「全国の売上」が見える。

## 2. マスキング関数の実装例 (Column Level)
UCでは SQLファンクションを定義して、それをカラムにバインドします。
```sql
-- マスキング用関数の定義
CREATE FUNCTION email_mask(email STRING)
  RETURN CASE WHEN is_account_group_member('hr_admins') THEN email ELSE '***@***' END;

-- カラムに適用
ALTER TABLE users ALTER COLUMN email SET MASK email_mask;
```

## 3. 行フィルタの実装例 (Row Level)
```sql
-- 北海道支店のデータだけを許可するフィルタ関数
CREATE FUNCTION region_filter(region STRING)
  RETURN is_account_group_member('global_admins') OR region = 'Hokkaido';

-- テーブルに適用
ALTER TABLE sales_data SET ROW FILTER region_filter ON (region);
```

---
## 🚨 The Incident: パフォーマンスの劣化
**事象**: 行フィルタを設定した途端、テーブルの読み取りが極端に遅くなった。
**原因**: フィルタ関数内で、外部の複雑なテーブルを何度も `JOIN` してチェックしている。
**対策**: 権限チェック用のマッピングテーブル（権限マスター）は、なるべく小さく保ち、`is_account_group_member()` などの高速な組み込み関数を多用してロジックをシンプルにします。

## 🎯 Mastery Mission: セキュリティ設計
**タスク**: 「給与テーブル」があります。
- `manager` グループの人は全員の金額が見えます。
- 一般社員は「自分の社員番号（`emp_id`）」の行だけが見えるようにしたいです。
- `current_user()` 関数（現在ログイン中のユーザー名を返す）を使って、どのような行フィルタ関数を作れば良いか、ロジックを日本語またはSQLで説明してください。
