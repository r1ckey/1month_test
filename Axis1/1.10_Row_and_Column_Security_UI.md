# [1.10] データガバナンス：Row-Level & Column-Level Security の実装

Unity Catalogでは、テーブル全体ではなく「特定の行」や「特定の列」に対して動的にアクセス制限をかける **「Fine-grained Access Control」** が可能です。

## 1. 行レベルセキュリティ (Row-Level Security)
「ユーザーの属性」や「グループ」に応じて、見せていい行だけをフィルタリングします。

### 実例: `region = 'Japan'` のユーザーには日本支社の売上しか見せない
1. フィルタ用関数 (Row Filter) を作成します。
2. テーブルにその関数を適用します。
```sql
ALTER TABLE sales SET ROW FILTER region_filter_func ON (region_column);
```

## 2. 列レベルセキュリティ (Column-Level Security / Column Masking)
「管理者にはクレジットカード番号（生データ）が見えるが、一般ユーザーには `****-****` と表示される」といった制御が可能です。

### 実例: `IS_ACCOUNT_GROUP_MEMBER('admin')` で分岐
1. マスキング用関数を作成します。
2. テーブルの特定の列にその関数を適用します。

> [!IMPORTANT]
> **動的ビュー (Dynamic Views) との違い**
> これまでは `CREATE VIEW` の中で `CASE WHEN` を書いて動的に制御していましたが、Unity Catalogではテーブルそのものに関数をアタッチできるため、管理が劇的に楽になりました。

---
**💡 実務のポイント:**
機密情報（PII: 個人特定情報）を扱う際は、基本設計段階で「どのカラムにマスキングをかけるか（データ保護要件）」を定義しなければなりません。
