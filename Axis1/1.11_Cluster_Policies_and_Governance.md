# [1.11] クラスターポリシー (Cluster Policies)：JSONによる徹底したコストガバナンス

「開発者が勝手に100台構成のクラスタを立ててしまい、一晩で100万円溶かした」
こうした悲劇を防ぐための、管理者必須の防衛策を学びます。

## 1. クラスターポリシーの役割
特定のユーザーやグループに対して、「作成できるクラスタのスペック」を制限します。
- **インスタンスタイプ制限**: 高価なGPUインスタンス（NCシリーズ等）を使わせない。
- **自動停止時間の短縮**: デフォルトの120分を30分に強制し、アイドル状態での課金を防ぐ。
- **スポットVMの強制**: 開発環境では安価なスポットVMの使用を義務付ける。

## 2. ポリシーのJSON定義例
```json
{
  "autotermination_minutes": {
    "type": "fixed",
    "value": 30,
    "hidden": true
  },
  "node_type_id": {
    "type": "allowlist",
    "values": ["Standard_DS3_v2", "Standard_DS4_v2"],
    "defaultValue": "Standard_DS3_v2"
  },
  "autoscale.max_workers": {
    "type": "range",
    "maxValue": 10,
    "defaultValue": 4
  }
}
```

## 3. 実務での使い分け
- **Personal Compute Policy**: 個人がアドホックな分析をするための、最小限のスペック。
- **Job Cluster Policy**: バッチ処理専用の、特定のスペックに固定された設定。
- **Gold Cluster Policy**: 経営層やBIツールからのみアクセスされる、高性能なパワーユーザ用。

---
## 🚨 The Incident: ポリシーが厳しすぎて開発が止まる
**事象**: ライブラリのインストールが必要なのに、ポリシーで `spark_conf` の変更が禁止されており、パッケージが追加できない。
**対策**: ポリシーを一律でガチガチにするのではなく、特定の信頼されたリードエンジニアだけはポリシーの適用を除外（Allow all）するなどの柔軟な運用を検討します。
