# [1.11] クラスタポリシー：JSONによるコスト制御とルール定義

開発者が自由に高性能なクラスタを立てられると、一晩で数百万円のコストが発生するリスクがあります。これを防ぐのが **「クラスタポリシー」** です。

## 1. クラスタポリシーの概要
クラスタ作成時の「ルール」を定義します。
*   **できること**: 特定のインスタンスタイプ（VMサイズ）のみ許可、オートスケールの最大値を制限、タグ付けの強制など。

## 2. JSONによるルール定義
管理者が作成し、特定のユーザーグループに付与します。
```json
{
  "spark_version": { "type": "fixed", "value": "13.3.x-scala2.12" },
  "node_type_id": { "type": "allowlist", "values": ["Standard_DS3_v2", "Standard_DS4_v2"] },
  "autoscale.max_workers": { "type": "range", "maxValue": 10 },
  "autotermination_minutes": { "type": "fixed", "value": 30, "isOptional": false }
}
```

## 3. 実務上のベストプラクティス
*   **タグ付けの強制**: 「Project Name」や「Cost Center」タグの入力を必須にすることで、Azureのコスト分析画面でどのプロジェクトがいくら使ったか一瞬で分かるようにします。

> [!IMPORTANT]
> **「fixed」と「allowlist」の使い分け**
> `fixed` はユーザーが変更不可、`allowlist` は選択肢の中から選ばせる、`range` は数値範囲を制限。これらを組み合わせて、自由度とコスト管理を両立させます。

---
**💡 運用設計のポイント:**
運用手順書では「全ユーザーは必ず『開発用ポリシー』を使用してクラスタを作成すること」と規定し、管理者以外の「自由なクラスタ作成権限」は剥奪するのが一般的です。
