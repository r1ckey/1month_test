# [1.05] AzureにおけるRBAC（Role-Based Access Control）設計

「誰が、どのリソースに対して、何ができるか」を管理するのがRBACです。Databricks自体の権限（Unity Catalog）とは別に、**Azureレイヤーでの権限設定**が必須です。

## 1. 3つの主要なロール
Azure Databricksを運用する上で頻出するロールは以下の通りです。

1. **Contributor (共同作成者)**: ワークスペースの作成、サブネットの設定、クラスタの作成などが可能。ただし、権限の付与（IAM変更）は不可。
2. **Owner (所有者)**: Contributorの全権限に加え、他者への権限付与が可能。
3. **Reader (閲覧者)**: リソースの存在を確認できるが、内容の変更やクラスタの起動は不可。

## 2. 実務での「権限最小化 (Least Privilege)」
開発者全員に `Contributor` を与えるのは危険です。
*   **インフラ担当**: `Contributor` (VNetやWSの管理)
*   **データエンジニア**: `Contributor` または特定のカスタムロール (Jobの実行やクラスタ作成のみ)
*   **アナリスト**: `Reader` + ワークスペース内の特定権限

## 3. ストレージへのアクセス権限
Databricks（またはそのマネージドID）がADLS Gen2の中身を読み書きするには、以下のロールが必要です。
*   **Storage Blob Data Contributor**: データの読み書き、削除が可能。
*   **Storage Blob Data Reader**: 読み取りのみ。

> [!CAUTION]
> **「Storage Account Contributor」では足りない！**
> 名前に騙されやすいですが、`Storage Account Contributor` は「ストレージアカウント自体の設定（キーの変更など）」を変更する権限であり、**「中のファイルの内容」を読む権限は含まれていません。** 必ず `Blob Data ...` 系のロールを付与してください。

---
**💡 チェックポイント:**
資格試験では、データにアクセスするために必要な最小限のロール名を選択させる問題が頻出します。
