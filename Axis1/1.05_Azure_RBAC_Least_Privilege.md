# [1.05] Azure RBAC 最小権限の原則 (Least Privilege) とカスタムロール

「権限が強すぎる」のは、データ消失における最大のリスクです。

## 1. 最小権限の原則 (Principle of Least Privilege)
「その作業に必要な最小限の権限だけを与える」こと。
- ❌ **管理者権限 (Owner)**: リソースの削除や権限の変更までできてしまう。
- ⭕️ **投稿者権限 (Contributor)**: リソースの作成・編集はできるが、権限変更はできない。
- 💎 **閲覧者権限 (Reader)**: 設定を見るだけ。

## 2. Databricks実務でよく使うロール
- **Azure Databricks Workspace Admin**: ワークスペース内のユーザー管理やクラスタ設定。
- **Storage Blob Data Contributor**: ADLSへの書き込み権限。
- **Private DNS Zone Contributor**: Private LinkのIP紐付け時に必要。

## 3. カスタムロールの作成
「Contributor」だと余計なリソース（VM等）まで作れてしまう、という厳しい環境では **「Custom Role (JSONで定義)」** を作ります。
```json
{
  "properties": {
    "roleName": "Databricks-Operator-Custom",
    "description": "Databricksのクラスタ操作とストレージ読み込みのみ許可",
    "assignableScopes": ["/subscriptions/xxxx"],
    "permissions": [
      {
        "actions": [
          "Microsoft.Databricks/workspaces/read",
          "Microsoft.Storage/storageAccounts/blobServices/containers/blobs/read"
        ],
        "notActions": [],
        "dataActions": [],
        "notDataActions": []
      }
    ]
  }
}
```

---
## 🚨 The Incident: 誰かがADLSを消してしまった！
**事象**: 本番用のストレージコンテナが、ある朝消滅していた。
**原因**: 開発者全員に「Owner」権限を与えていたため、一人の操作ミス（削除）が防げなかった。
**対策**: リソースには **「リソースロック (Resource Lock)」** をかけ、権限は `Contributor` 以下に、さらに作業用IDと閲覧用IDを分離します。
