# [1.04] Entra ID Service Principals と Managed Identities (MSI)

データパイプラインを自動化（Job実行）する際、個人のアカウント（User Identity）を使うのはNGです。退職時にジョブが止まるからです。実務では「サービス用のアカウント」を使います。

## 1. サービスプリンシパル (Service Principal)
Azure Entra ID (旧AD) における「アプリケーション用のアカウント」です。
*   **仕組み**: クライアントIDとシークレット（パスワード）を使って認証します。
*   **Databricksでの用途**: 外部システム（Azure Data Factory等）からDatabricks APIを叩く際によく使われます。

## 2. マネージドID (Managed Identity / MSI)
Azureサービス（VMやDatabricks）自体に、IDを直接付与する最強に安全な方法です。
*   **メリット**: **パスワード（シークレット）の管理が不要**です。Azure内部で自動的に認証が行われます。
*   **Databricksでの用途**: 
    1. Unity Catalogがストレージ（ADLS Gen2）にアクセスするための「Storage Credential」として使用。
    2. クラスタにアタッチし、DBXからAzure Key Vaultなどの他サービスへ認証なしでアクセス。

> [!TIP]
> **シークレットレスの追求**
> セキュリティ事故の多くは「ソースコードに書き込まれたパスワードの漏洩」です。Managed Identityを使えば、コード内にパスワードを書く必要が一切なくなるため、可能な限りMSIを選択するのがプロの設計です。

---
**💡 実務のポイント:**
基本設計書では、どの接続（Databricks ➡️ ADLSなど）でどのアイデンティティ（SPNまたはMSI）を使うかをマトリクス図で明記する必要があります。
