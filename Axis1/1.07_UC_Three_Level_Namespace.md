# [1.07] UCの三段階名前空間（Three-Level Namespace）設計

「データの住所」をどう決めるか。これが将来の保守性を左右します。

## 1. 3-Level Namespace 構造
今までの `table_name` だけの指定から、以下の形式に変わります。
`catalog_name . schema_name . table_name`

1.  **Catalog**: プロジェクトや部門単位（例： `sales_dept`, `marketing_dev`）。
2.  **Schema (Database)**: 業務ロジックやメダリオン層（例： `bronze`, `gold_reporting`）。
3.  **Table / View**: 実際のデータ。

## 2. 実務でのカタログ設計パターン
- **環境分離型**: `main` (本番), `dev` (開発), `stg` (検証) というカタログを作る。
- **組織分離型**: `hr_dept`, `finance_dept` のように部署ごとにカタログを分け、お互いのカタログは原則見えないようにする。

## 3. SQLでのアクセス例
```sql
-- 他のカタログにあるデータを結合する（カタログ名から指定可能！）
SELECT * 
FROM dev_catalog.bronze.users u
JOIN main_catalog.gold.master_codes m ON u.id = m.user_id
```

## 🚨 The Incident: `default` スキーマの乱用
**事象**: 全てのテーブルを `main.default.xxxx` に作ってしまい、テーブルが1000個を超えてどれが何だか分からなくなった。
**解決策**: `default` スキーマはあくまで一時利用に留め、業務単位やメダリオンの層（Bronze/Silver/Gold）に基づいた明確なスキーマ設計を最初に行います。
**教訓**: 設計書（Axis 3で学びます）で、カタログ名の命名規則をガチガチに固めることが、アーキテクトの最初の仕事です。

- 以下の3つのカタログ名を考えてください。
