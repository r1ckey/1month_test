# [1.07] UCの階層構造（Catalog > Schema > Table）と名前空間

Unity Catalogは **「3レベルの名前空間 (3-level namespace)」** を採用しています。これが全てのデータ参照の基本となります。

## 1. 3レベルの階層

1.  **Catalog**: メタデータ管理のトップ。データアセットの所有権やアクセス権の主要な境界。
2.  **Schema (Database)**: カタログ内の論理的なグループ。テーブルやビューをまとめます。
3.  **Table / View**: 最小のデータ単位。

### SQLでの参照方法
```sql
SELECT * FROM <catalog_name>.<schema_name>.<table_name>
```

## 2. メリット
*   **名前の衝突回避**: ワークスペース間で同じテーブル名を使っても、カタログ名が違えば干渉しません。
*   **環境分離**: `dev_catalog`, `prod_catalog` のようにカタログレベルで物理的なアクセス制限をかけることができます。

> [!TIP]
> **デフォルト設定の活用**
> ワークスペースレベルで「デフォルトカタログ」を設定しておくことで、SQL内でカタログ名を省略（例: `schema.table`）して記述することも可能です。

---
**💡 実務のヒント:**
基本設計書において「名前付け規則（Naming Convention）」は非常に重要です。`env_projectname_system` のようなカタログ命名ルールを定義し、それを踏襲するよう徹底します。
