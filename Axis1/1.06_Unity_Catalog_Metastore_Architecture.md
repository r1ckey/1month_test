# [1.06] Unity Catalog (UC) の全体アーキテクチャ：なぜメタストア分離が必要か

「テーブルがバラバラに存在するカオス」を終結させ、全社的なガバナンスを敷くための新基盤を理解します。

## 1. 従来（Hive Metastore）の限界
Unity Catalog以前は、ワークスペースごとに 「Hive Metastore」 が存在していました。
- **問題点**: ワークスペースAで作ったテーブルは、ワークスペースBからは直接見えない（または複雑な同期が必要）。
- **問題点**: 権限管理が Azure RBAC やIAMに依存しすぎており、SQL的な `GRANT` でのきめ細かな制御が難しかった。

## 2. Unity Catalog の核心：メタストアの一元化
UCでは、特定のリージョン（例：Japan East）に **「1つのメタストア」** を作成し、そこに複数のワークスペースを紐づけます。
- **共有**: ワークスペースを跨いで、全く同じテーブル名でデータにアクセス可能。
- **監査**: 「誰が・いつ・どのデータを見たか」を全ワークスペース分まとめて取得可能。

## 3. 実務でのセットアップ（前提条件）
アーキテクトとして、以下の3つを揃える必要があります。
1.  **Unity Catalog用ADLSコンテナ**: UCが管理用データを置くための専用ストレージ。
2.  **マネージドID**: DatabricksがそのADLSにアクセスするための鍵。
3.  **アカウント管理者権限**: Databricksの「Account Console」に入れる権限。

---
## 🚨 The Incident: メタストアは「全リージョンで1つ」ではない
**事象**: 米国リージョンと日本リージョンのワークスペースで、同じUCメタストアを共有しようとしたが失敗した。
**解決策**: UCメタストアは **リージョン単位** です。日本にあるメタストアを米国のワークスペースに紐づけることはできません（2024年現在）。
**教訓**: マルチリージョン展開するプロジェクトでは、リージョンごとにメタストアを設計し、必要に応じて「Delta Sharing」を用いたデータ共有を検討します。
