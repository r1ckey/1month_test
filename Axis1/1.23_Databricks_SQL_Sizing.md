# [1.23] Databricks SQL ウェアハウスのサイジングと並列処理

BIツールからのクエリが遅い原因の多くは、ウェアハウスのサイズ設定ミスです。

## 1. T-Shirt サイジング
ウェアハウスのサイズは `2X-Small` から `4X-Large` まで、Tシャツのサイズのように選ぶだけです。サイズが上がるごとに、裏側のクラスタ性能が倍々になっていきます。

## 2. サーバーレス vs プロビジョニング済み
*   **Serverless SQL Warehouse**: 起動が早く、アイドル時に課金されない。
*   **Classic / Pro SQL Warehouse**: 従来のVMクラスタ管理。

## 3. スケーリング戦略
*   **アップスケーリング (Vertical)**: サイズを「Small」から「Large」に変更する。複雑な「巨大な1つのクエリ」を早くしたい時に有効。
*   **ダウンスケーリング (Horizontal)**: 最大クラスタ数を増やす。多数のユーザーが「同時に多くのクエリ」を投げる際の待ち時間を減らしたい時に有効。

> [!TIP]
> **インテリジェントな自動停止 (Auto-Stop)**
> ユーザーが離席してクエリが止まった際、数分（デフォルト10分など）で自動停止するように設定し、無駄なDBU消費を防ぎます。

---
**💡 性能設計のヒント:**
「このダッシュボードの表示が遅い！」というクレームが来たら、まずウェアハウスのサイズの引き上げ（垂直スケール）を検討してください。同時に投げられてタイムアウトしているなら、クラスタ数の増加（水平スケール）です。
