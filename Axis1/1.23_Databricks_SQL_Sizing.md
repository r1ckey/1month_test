# [1.23] Databricks SQL ウェアハウスのサイジング(Sizing)と負荷分散

「SQLの実行が遅いのは、エンジンの設定ミスか？それともSQLの書き方か？」

## 1. ウェアハウスのサイズ (Size)
- **2X-Small (最小) 〜 4X-Large (最大)** まで存在します。
- **性質**: サイズを上げると、計算リソース（メモリとCPU）が倍増します。大規模なテーブル（数十TB）をスキャンする際の「瞬発力」を高めます。

## 2. スケーリング (Auto-Scaling)
- **Min Clusters / Max Clusters**: 同時に100人のユーザーが接続してきても、自動でクラスタを「横に増設」して待ち時間を減らします。
- **実務**: 参照専用のダッシュボードなら `Max Clusters` を多めに、開発者用なら `1` に固定してコストを抑えます。

## 3. Serverless vs Pro / Classic
- **Pro**: `V-Order` や `Photon` が有効な高性能版。
- **Serverless**: 待機時間ゼロの最新版。実務では **Serverless** を強く推奨します。

---
## 🚨 The Incident: 特定のクエリだけが異常に遅い
**事象**: ウェアハウスを Large に上げても、特定の `JOIN` クエリだけが15分以上かかる。
**原因**: スケールアップ（Size）で解決する問題ではなく、データの並び順（Z-Order）や結合ロジック（Broadcasting）に問題がある。
**教訓**: 「金（リソース）で解決できる遅延」と「エンジニアリング（設計）でしか解決できない遅延」を切り分けましょう。
