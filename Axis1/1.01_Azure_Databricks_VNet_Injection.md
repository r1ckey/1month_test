# [1.01] Azure Databricksの全体像とVNet構築：実務ネットワーク設計の極意

単にポータルでデプロイするだけではなく、企業の「堅牢なインフラ」として機能させるための設計を学びます。

## 1. VNet Injection のアーキテクチャ深掘り
実務ではデフォルトのマネージドVNet（Databricksが勝手に作るVNet）はほぼ使いません。自前のVNet（VNet Injection）を使う理由は以下の3点に集約されます。

- **オンプレミス接続**: ExpressRoute経由で社内DBからデータを抜くには、社内VNetに属している必要があります。
- **IPアドレスの枯渇回避**: クラスタ規模に応じた柔軟なIPレンジ設計。
- **セキュリティ・アプライアンス**: 全ての通信を Azure Firewall やプロキシサーバ経由で強制するための UDR（ユーザー定義ルート）の設定。

## 2. 実務のサブネット・サイジング（計算式）
Workerノード1台につき、**2つのIPアドレス**が必要です（Private SubnetとPublic Subnetから1つずつ）。

### 💡 設計計算例
- 最大クラスタサイズ: 50ノード
- 同時実行ジョブ数: 2
- 予備（バッファ）: 20%
- **必要IP数** = (50 * 2 + バッファ) * 2（サブネット数） = 約240個
- **結論**: **/24 (251使用可能IP)** のサブネットを2つ用意するのが「安全牌」です。/26だと、大規模プロジェクトでは数ヶ月でIPが枯渇し、VNetの作り直し（地獄）が発生します。

## 3. Terraform による定義（Snippet）
インフラエンジニアとして、これをコードで定義できる必要があります。
```hcl
resource "azurerm_subnet" "public" {
  name                 = "dbx-public-sn"
  resource_group_name  = azurerm_resource_group.rg.name
  virtual_network_name = azurerm_virtual_network.vnet.name
  address_prefixes     = ["10.0.1.0/24"]
  delegation {
    name = "databricks-delegation"
    service_delegation {
      name = "Microsoft.Databricks/workspaces"
      actions = ["Microsoft.Network/virtualNetworks/subnets/join/action", "Microsoft.Network/virtualNetworks/subnets/prepareNetworkPolicies/action"]
    }
  }
}
```

## 🚨 The Incident: もしサブネットを小さく設定してしまったら？
**事象**: 業務拡大によりクラスタを100台に増やそうとしたが、`SubnetFull` エラーで立ち上がらない。
**解決策**: 既存のサブネットは拡張できません。新しいVNet/Subnetを作り、Databricksワークスペースを再起動（または新規作成して移行）する必要があります。
**教訓**: 「最初は小さく」はインフラ設計では禁忌です。余裕を持ったレンジを最初から確保しましょう。
