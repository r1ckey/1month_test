# [1.01] Azure Databricksの全体像とVNet構築

Azure Databricksを実務で導入する際、最も最初にぶつかる壁が「ネットワーク（VNet）設計」です。デフォルトの「マネージドVNet」ではなく、企業のセキュリティポリシに合わせた「VNet Injection（自前のVNetへの注入）」が標準となります。

## 1. VNet Injectionとは？
Azure Databricksの「Compute Plane（計算リソース）」を、ユーザーが管理する既存のAzure Virtual Network (VNet) 内にデプロイする手法です。

### なぜ必要なのか？
1. **サービス終端 (Service Endpoints/Private Links)**: オンプレミスや他のAzureサービス（SQL DB等）に閉域網で接続するため。
2. **IPアドレス管理**: 企業内のIPアドレス体系に組み込み、重複を避けるため。
3. **カスタムルート (UDR)**: 全てのトラフィックをFirewall（Azure Firewall等）経由でインターネットへ出したい場合。

## 2. サブネットの設計ルール
VNet Injectionには、**2つの専用サブネット**が必要です。
*   **Public Subnet (ホストサブネット)**: Workerノード間、およびControl Planeとの通信に使用。
*   **Private Subnet (コンテナサブネット)**: Worker内のコンテナ間通信に使用。

> [!IMPORTANT]
> **サブネットのサイズ選び (CIDR)**
> 各サブネットは同じサイズである必要があり、少なくとも /26 (64個のIP) が必要ですが、実務では将来のオートスケールを考慮し ** /24 (256個のIP) ** 以上を確保するのが定石です。

## 3. ネットワークセキュリティグループ (NSG)
Databricksはデプロイ時にこれらのサブネットにNSGを自動適用（または更新）します。
*   **Inbound**: Worker同士が通信できるように特定のポート（22, 5557など）が許可されます。
*   **Outbound**: Azure StorageやDatabricks Control Planeへの通信が許可されます。
