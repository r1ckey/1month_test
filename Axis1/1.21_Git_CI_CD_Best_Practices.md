# [1.21] Git 連携と GitHub Actions による CI/CD ベストプラクティス

「手動デプロイ」はヒューマンエラーの温床です。全自動の配送ライン（CI/CD）を構築します。

## 1. ワークスペースの Git 連携 (Repos/Git Folders)
Databricks内部で直接Gitリポジトリをクローンし、開発を行うことができます。
- **メリット**: `git checkout`, `git commit` がUI上から可能。
- **実務ルール**: メインブランチ（`main` / `master`）は直接編集禁止にし、PR（プルリクエスト）経由でのマージを必須にします。

## 2. GitHub Actions による自動デプロイ
エンジニアがPRをマージした瞬間、自動的に本番環境へ反映される仕組み。
```yaml
# GitHub Actions の定義例 (.github/workflows/deploy.yml)
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: databricks/setup-cli@main
      - run: databricks bundle deploy -t prod
        env:
          DATABRICKS_HOST: ${{ secrets.PROD_HOST }}
          DATABRICKS_TOKEN: ${{ secrets.PROD_TOKEN }}
```

## 3. シークレット（環境変数）の保護
GitHubの `Settings > Secrets` に、本番環境の `DATABRICKS_TOKEN` を保存します。
- **注意**: ログ（出力）にトークンが表示されないよう、GitHub Actions側でのマスキング設定も重要です。

---
## 🚨 The Incident: 開発中のコードが本番で走ってしまった
**事象**: 開発者が自分のブランチで作業していたつもりが、誤って本番用ブランチにデプロイ命令を投げてしまい、未完成のロジックが本番Jobに上書きされた。
**対策**: GitHubの **「Branch Protection Rules」** を使い、本番ブランチへのプッシュは特定の承認者（レビュー）がいない限りブロックするように設定します。
