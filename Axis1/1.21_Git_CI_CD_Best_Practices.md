# [1.21] Git連携・GitHub Actions による CI/CD分離戦略

データ分析の現場で最も多いトラブルの一つが「誰かが誤って本番環境のNotebookを上書きしてしまった」という事故です。これを防ぐのが **Git連携** と **CI/CD** です。

## 1. Databricks Git フォルダ (Repose)
Databricksの画面上で、GitHubやAzure DevOpsのリポジトリを直接クローンできる機能です。
*   **特徴**: Notebook上の変更を、そのまま `git commit` & `push` できます。

## 2. 開発と本番の分離 (Isolation)
*   **開発環境**: エンジニアが自由にNotebookを編集し、テスト的な実行を行う。
*   **本番環境**: **人間はNotebookを触れない**設定にします！コードの更新は GitHub Actions 等のCIツールからのみ、自動デプロイされるように制限します。

## 3. GitHub Actions のワークフロー例
```yaml
# .github/workflows/deploy.yml
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install Databricks CLI
        run: curl -fsSL https://raw.githubusercontent.com/databricks/setup-cli/main/install.sh | sh
      - name: Deploy Asset Bundle
        run: databricks bundle deploy -t prod # 承認されたコードのみを本番へ！
        env:
          DATABRICKS_HOST: ${{ secrets.PROD_HOST }}
          DATABRICKS_TOKEN: ${{ secrets.PROD_TOKEN }}
```

> [!IMPORTANT]
> **プルリクエスト (PR) ベースの開発**
> 「他人のレビューを受けて承認（Approve）されたコードだけが、自動的に本番Jobに反映される」という流れを作ることが、エンタープライズ品質への第一歩です。

---
**💡 運用設計のポイント:**
運用マニュアルには「本番環境での直接編集を禁止する」というルールと、万が一の緊急修正が必要な場合の「エスカレーションフロー」を必ず明記します。
