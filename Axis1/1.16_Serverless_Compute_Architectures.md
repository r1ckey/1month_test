# [1.16] Serverless SQL Warehouse と Serverless Jobs のアーキテクチャ

これまでは、SQLを叩くためにも事前に「VMクラスタ」を立ち上げて待つ必要がありました。これを完全に抽象化したのが **「サーバーレス (Serverless)」** モデルです。

## 1. サーバーレスの正体
サーバーレスと言っても、裏側ではVM（計算リソース）が動いています。ただし、そのVMは**「Databricks側（Microsoft管理サブスクリプション内）」**で事前にプールされており、起動命令を出した瞬間に数秒でアタッチされます。

## 2. Serverless SQL Warehouse
BIツール（TableauやPowerBI）からSQLを投げる際に使用します。
*   **メリット**:
    1.  **起動が数秒**: 通常のウェアハウス（数分）より圧倒的に早い。
    2.  **管理不要**: パッチ適用やVNet設計の複雑さを考えなくて良い。
    3.  **コスト**: 使っていない時は完全に0円（アイドル時の課金なし）。

## 3. Serverless Jobs
データエンジニアのバッチ処理（NotebookやPythonファイル）をサーバーレスで実行します。
*   **メリット**: インフラの専門知識がないアナリストでも、ワンクリックでスケーラブルなバッチ処理をデプロイ可能。

> [!CAUTION]
> **ネットワーク境界の変化**
> サーバーレスは「Databricks管理のサブスクリプション」で動くため、自分たちのVNet内の閉域なADLS Gen2にアクセスするには、**「サーバレスネットワーク接続 (Serverless Connectivity)」**の設定が別途必要になります。

---
**💡 アーキテクチャ設計のコツ:**
「常時大量のクエリが来るBI用」には、従来のプロビジョニング済みウェアハウスの方が安くなる場合もあります。変動の激しい「散発的なクエリ」にこそ、サーバーレスの真価が発揮されます。
