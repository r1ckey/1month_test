# [1.20] DABs を用いた databricks.yml 構成とリソースデプロイ

DABsの心臓部は、プロジェクトのルートに置かれる **`databricks.yml`** ファイルです。

## 1. `databricks.yml` の構造
このファイルに、「どの環境（Dev/Prod）に」「どんなJobを」デプロイするかを記述します。

```yaml
bundle:
  name: my_data_project

targets:
  dev:
    workspace:
      host: https://adb-dev.azuredatabricks.net
  prod:
    workspace:
      host: https://adb-prod.azuredatabricks.net

resources:
  jobs:
    daily_etl_job:
      name: "Daily ETL Pipeline"
      tasks:
        - task_key: main_processing
          notebook_task:
            notebook_path: ./src/main_notebook.ipynb
```

## 2. 変数の活用
環境ごとに異なる値（クラスタサイズなど）を `${var.xxx}` という形式で外部注入できます。

## 3. デプロイコマンド
```bash
# まず検証 (バリデーション)
databricks bundle validate

# 開発環境へデプロイ
databricks bundle deploy -t dev

# ジョブの実行テスト
databricks bundle run daily_etl_job
```

> [!IMPORTANT]
> **GitHub Actions との親和性**
> `databricks bundle deploy` コマンドはCI/CDツール（GitHub Actions）から実行するのに最適に設計されています。これにより、「GitHubにコードをマージした瞬間に、自動でDatabricks上のJobが更新される」というフローが完成します。

---
**💡 実務のポイント:**
これまでの手動Job作成は、作成した本人しか設定を知り得ない「ブラックボックス」になりがちでした。DABs（YAML）を使うことで、Job設定そのものが「コード」としてレビュー可能になり、属人化を劇的に破壊できます。
