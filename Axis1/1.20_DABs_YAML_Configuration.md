# [1.20] DABs を用いた databricks.yml 実装構成：環境スイッチの自動化

具体的に、どうやって「開発用」と「本番用」のインフラをコードで切り替えるか。

## 1. `databricks.yml` の基本構成
このファイル一つで、デプロイ先の環境（Targets）を定義します。

```yaml
bundle:
  name: sales-pipeline

targets:
  dev:
    workspace:
      host: https://adb-dev.azuredatabricks.net
    mode: development
  prod:
    workspace:
      host: https://adb-prod.azuredatabricks.net
    mode: production
```

## 2. 環境ごとの変数の使い分け
「開発用DB」と「本番用DB」の接続先を変えたい時、DABsの **Variables** 機能が威力を発揮します。
```yaml
variables:
  storage_path:
    default: "dbfs:/dev/data"

targets:
  prod:
    variables:
      storage_path: "dbfs:/prod/data"
```
Notebook内では、この `storage_path` を環境変数のように参照して、コードを一切変えずに環境を切り替えられます。

## 3. 実務でのデプロイコマンド
```bash
# 開発環境へデプロイして、即座にJobを実行してテストする
databricks bundle deploy -t dev
databricks bundle run my_job -t dev
```

---
## 🚨 The Incident: 本番環境への誤デプロイ
**事象**: 開発環境のつもりでコマンドを叩いたが、実は本番環境のトークンを設定しており、本番のJob設定を書き換えてしまった。
**対策**: DABsの `production` モードを有効にします。このモードでは、デプロイ時に「本当に本番へ適用して良いか？」の確認が表示されたり、破壊的な変更に制限がかかるようになります。
