# [1.25] Azure Private DNS ゾーンと閉域における名前解決(DNS)の罠

Private Linkを組んだとき、最もエンジニアを悩ませる「名前解決」の闇を照らします。

## 1. Private Link と DNS の仕組み
通常、`adb-xxxx.azuredatabricks.net` はグローバルなDNSサーバが「パブリックIP」を返します。
しかし、Private Linkを有効にした場合、**「プライベートIP（例：10.1.2.5）」** が返ってこなければなりません。

## 2. 必須となる DNS ゾーン名
以下のプライベートDNSゾーンをAzure上に作成し、自社VNetにリンクさせる必要があります。
- `privatelink.azuredatabricks.net`
- `privatelink.blob.core.windows.net` (ADLS用)
- `privatelink.dfs.core.windows.net` (ADLS gen2用)

## 3. 実務の罠：オンプレミスからの名前解決
ここが最大の難所です。
- **現象**: 自分のPC（VPN接続）から `ping adb...` すると、依然としてパブリックIPが返ってきてしまい、Private Linkを通れない。
- **解決策**: Azure側に **「DNS Private Resolver」** を立て、オンプレミスのDNSサーバからAzure側のPrivate DNSを検索（転送）するように設計します。

---
## 🚨 The Incident: ストレージだけ繋がらない！
**事象**: Databricksの画面は見れるが、データを読み込もうとすると `Connection Refused` になる。
**原因**: `blob.core.windows.net` のPrivate DNSゾーンは作ったが、`dfs.core.windows.net` のゾーンを忘れていた（ADLS Gen2は両方のエンドポイントを使う場合があります）。
**教訓**: ストレージ系は「Blob」と「DFS」の両方のPrivate DNSレコードを確認しましょう。
