# [3.16] テストデータ合成 (Synthetic Data) とマスキングの実務

「本番データを使えない」という制約下で、いかにリアルなテストを行うか。

## 1. なぜ本番データを使っていけないのか
- **セキュリティ**: 開発環境の管理レベルは本番より低く、個人情報漏洩のリスクがある。
- **再現性**: 本番データは刻々と変わるため、バグ修正の確認が難しい。

## 2. テストデータ作成の 3つの手法
1.  **マスキング (Masking)**: 本番データをコピーし、氏名を「ダミー太郎」に、クレカ番号を `***` に上書きする。
2.  **サンプリング (Sampling)**: 10億件のうち、1万件だけを抽出する。
3.  **合成 (Synthesis)**: `faker` などのライブラリを使い、本物らしい分布を持った架空データを一から生成する。

## 3. 実務での「最重要データ」の作り込み
「ただのテスト用文字列」ではダメです。
- **日付**: 未来の日付、過去すぎる日付、うるう年。
- **金額**: 0円、マイナスの値、100億円などの極端な数値。
- **特殊文字**: 苗字に「﨑」や「𠮷」が入っているケース（文字化けテスト）。

---
## 🚨 The Incident: 本番データが開発者に丸見え！
**事象**: 本番トラブルの調査のため、一次的に開発者に本番DBの閲覧権限を与えたが、その権限を消し忘れて、数ヶ月間個人情報が見放題になっていたことが監査で発覚。
**対策**: 直接本番を触らせるのではなく、特定の **「調査用マスキングツール」** を通じたデータ抽出のみを許可するフローに固定します。
