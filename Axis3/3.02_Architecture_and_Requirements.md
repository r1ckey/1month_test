# [3.02] 構成図と要件定義書の落とし込み：ふわっとした要望を技術に翻訳する

「ビジネスのやりたいこと」を「Azureのアイコン」に変換する翻訳術。

## 1. 要件定義（Requirement Definition）の分解
顧客は「いい感じにデータを可視化したい」としか言いません。それを以下に分解します。
- **データ量**: 1日何GB？ 保存期間は何年？
- **鮮度**: リアルタイム（秒）？ それとも翌朝（バッチ）？
- **ユーザー**: 誰が見る？ 同時に何人がアクセスする？

## 2. 構成図（Logical Architecture）の3点セット
1.  **コンテクスト図**: システムの外側の人間や他システムとの関係。
2.  **論理構成図**: Azure上のサービスの繋がり（VNet, SQL DB, Databricks）。
3.  **データフロー**: データがどう加工されていくかの流れ。

## 3. 実務での「トレードオフ分析」
- **要望**: 「安く、かつ、一瞬で結果が出るようにしてほしい」
- **回答**: 「一瞬で出すには Serverless SQL の Xサイズ が必要で、月額 ◯万円 かかります。予算に合わせるなら 1分 待つ設定にしましょう」
このように、**「金額 vs パフォーマンス」** の天秤を提示することがアーキテクトの仕事です。

## 🚨 The Incident: 隠れた非機能要件による炎上
**事象**: バッチジョブは完成したが、実は「深夜2時までに終わらなければならない」という制約が後から発覚。データ量が増えて 3時 にずれ込み、翌朝の業務が止まった。
**対策**: 要件定義の初期段階で「SLA（サービス品質保証）」と「バッチ突き抜け時の対応方針」を文書化し、顧客と合意します。
