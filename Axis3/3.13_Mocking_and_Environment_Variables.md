# [3.13] モック (Mock) 戦略と環境変数の活用

「外部システム（DBやAPI）が止まっていても、テストを完遂させる」

## 1. モック (Mocking) とは
テスト中に、本物の外部サービスを呼ぶ代わりに、あらかじめ決まった偽の返り値（Fake）を返す代用品のこと。
- **Databricks での用途**: 
    - Azure Key Vault からパスワードを取る部分。
    - 外部の REST API から JSON を取る部分。

## 2. `unittest.mock` の活用例
```python
from unittest.mock import patch

# 「APIを叩く関数」をテスト中だけ乗っ取る
with patch('requests.get') as mocked_get:
    mocked_get.return_value.json.return_value = {"status": "ok"}
    # ここで本当の関数を呼んでも、APIは叩かれず {"status": "ok"} が返ってくる
```

## 3. 環境変数 (Environment Variables) による切り替え
「本番なら本物のDBへ、テストならローカルのファイルへ」という切り替えを、コードを書き換えずに実現します。
- `os.environ.get("ENV") == "TEST"` なら Mock を使う、といったロジックを共通部品として定義します。

---
## 🚨 The Incident: テスト用のデータが本番 DB を書き換えた
**事象**: 結合テストを実行した際、接続先設定が「本番」のままになっており、テストデータが本番の顧客テーブルを汚染してしまった。
**対策**: 
1.  接続用 ID に「削除権限」を与えない（ Axis 1.05 ）。
2.  コード内で **「接続先 URL が localhost または stg 以外の場合はエラーを出す」** 防護壁（ガード句）をテスト時のみ有効にします。
