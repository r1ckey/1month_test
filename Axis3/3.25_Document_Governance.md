# [3.25] ドキュメントの構成管理・レビュー：知識の腐敗を防ぐ

「古いドキュメントは、無いよりも害悪である」

## 1. 知識の腐敗 (Knowledge Rot)
システムが変わったのに、設計書が更新されない状態。
- **末路**: 新しいメンバーが間違った情報を信じてバグを出し、その修正のためにさらに古い設計書を見……という負のスパイラル。

## 2. 実務の解：Markdown によるドキュメント管理 (Docs-as-code)
Word や Excel ではなく、**Markdown ファイルとして Git で管理** します。
- **メリット**:
    - コードと設計書を同時に Pull Request できる。
    - 「誰が・いつ・どこを」変えたか、差分（Diff）が 1行 単位でわかる。
    - **STUDY2** 自体も、この Docs-as-code で構築されています。

## 3. レビュー文化の醸成
「コードレビュー」はするのに「設計レビュー」をしないプロジェクトは多いです。
- 設計段階で 1分 考えることは、開発段階での 1時間 の手戻りを防ぎます。

---
## 🚨 The Incident: 名人しか知らない「裏設定」
**事象**: リードエンジニアが退職した途端、誰もシステムの仕組みを説明できなくなり、保守費用が 2倍 に膨らんだ。
**対策**: 「忙しくて書けません」は許さない文化を作ります。**「ドキュメント（DABsの定義やMarkdown）が更新されていない Pull Request は、絶対にマージしない」** という鉄の掟をプロジェクトに導入します。
