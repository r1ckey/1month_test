# [3.11] テスト仕様とスコープの定義：同値分割と境界値分析

「どこまでテストすれば『安心』と言えるのか」

## 1. テストの 3階層 (Test Levels)
1.  **単体テスト (Unit Test)**: 1つの関数や 1つの変換ロジックが正しいか。
2.  **結合テスト (Integration Test)**: Aテーブルから Bテーブルへの流れが正しいか。
3.  **システムテスト / 受入テスト (ST / UAT)**: 深夜バッチが最後まで走り、ダッシュボードが更新されるか。

## 2. 賢く網羅する「テスト技法」
全パターンを試すのは不可能です。以下の 2つ を使い、最小の手間で最大のバグを見つけます。
- **同値分割 (Equivalence Partitioning)**: 
    - 「1〜100なら合格、それ以外は不合格」という仕様なら、`50` (合格圏) と `150` (不合格圏) の 2つ を試せば、その範囲の代表として十分です。
- **境界値分析 (Boundary Value Analysis)**: 
    - バグは「境目」に潜んでいます。`0`, `1`, `100`, `101` の 4点 をテストします。

## 3. 実務での「テスト仕様書」
Excel や Wiki に以下を記述します。
- **前提条件**: テスト前のデータ状態。
- **テスト操作**: どのノートブックを動かすか。
- **期待結果**: 保存されたテーブルの件数、特定カラムの値。

## 🚨 The Incident: 休日・うるう年での計算ミス
**事象**: 本番稼働から 1ヶ月、`2024-02-29` （うるう年）の集計で Job が異常終了した。
**原因**: プログラム内で「1年は 365日」と固定して計算していたため。
**教訓**: 日付計算が絡むロジックでは、テストデータに必ず「末日（30, 31日）」や「うるう年」を含めるよう、境界値分析を徹底します。
