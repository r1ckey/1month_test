# [3.22] Triage（切り分け）：ログ追跡と原因特定の 5ステップ

「なぜ壊れたか」ではなく、「どこまで壊れていないか」を探る。

## 1. ログ追跡の優先順位
1.  **Workflows の出力**: どの Task でコケたかを確認。
2.  **Spark Driver ログ**: プログラムの文法エラーや、NULL参照を確認。
3.  **Spark Executor ログ**: 1台のマシンだけの異常（メモリ不足等）を確認。
4.  **Audit Logs (Axis 1.15)**: 誰かが設定をいじっていないかを確認。

## 2. 原因特定の 5ステップ (The 5 Whys)
- **現象**: データが届かない。
- 1. **なぜ？**: ジョブがタイムアウトした。
- 2. **なぜ？**: 保存先のテーブルがロックされていた。
- 3. **なぜ？**: 別のクリーンアップジョブが終わっていなかった。
- 4. **なぜ？**: ...

## 3. 実務スキル： `DESCRIBE HISTORY`
Delta テーブルがおかしい時は、まずこの一撃で「誰が最後に何を書き込んだか」の履歴を見ます。

---
## 🚨 The Incident: 直近の変更だけが原因とは限らない
**事象**: 今朝のリリース後に Job が落ちたので、リリース分を戻したが、それでも治らなかった。
**真因**: 実は 1週間前からソースデータの形式が変わっており、今日溜まりに溜まったエラー行が閾値を越えて爆発しただけだった。
**教訓**: リリース履歴だけでなく、**ソースデータの「傾向（NULL率や最大値）」** の変化も切り分けの対象に含めましょう。
