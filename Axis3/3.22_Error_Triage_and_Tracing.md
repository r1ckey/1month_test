# [3.22] Triage（トリアージ・切り分け）：ログの追跡手順（DBX Logs vs Airflow Logs）

「どこで死んでいるか」を特定するための、データエンジニアの探偵業務です。

### 1. ログの階層構造
1.  **Orchestrator Logs (Airflow / DBX Jobs)**: 「ジョブの起動に失敗したか、中身のNotebookで死んだか」の切り分け。
2.  **Spark UI / Driver Logs**: 「計算そのものが失敗したのか（OOM等）」「関数がエラーを吐いたのか」を確認。
3.  **ADLS Gen2 Logs**: 「そもそもファイルが無かったのか」「権限エラーで読めなかったのか」。

### 2. トリアージの極意
まず「**自分のせい（コード）か、相手のせい（インフラ/ソース）か**」を秒速で判断します。
*   相手のせいであれば、速やかにその部門に調査を依頼し、自分たちの手を空ける（時間の節約）のがプロの立ち回りです。

---
**💡 実務のアドバイス:**
エラーログをググる際は、「日付」や「ファイルパス」といった自分に固有の情報（UUIDなど）を抜き、エラーメッセージの「核」となる部分だけで検索するのがコツです。
