# [3.04] データ連携設計：マッピング表の作成とETL定義

「どの列をどう加工して、どこに入れるか」という極めて緻密な設計。

## 1. データマッピング定義書 (Mapping Table)
ETL実装のバイブルです。以下の列を Excel や Wiki で作成します。
- **項番**: 管理用 ID。
- **ソース項目名**: 基盤となるテーブル/ファイルの列名。
- **変換ロジック**: 物理名での計算式（例： `COALESCE(amount, 0) * 1.1`）。
- **ターゲット項目名**: 保存先の列名。
- **型・桁数**: NULL 許可、主キー設定など。

## 2. 変換ロジックの「日本語」問題
エンジニアが一番苦労するのが「日本語の曖昧さ」です。
- ❌ **悪い例**: 「空白だったら0にする」
- ⭕️ **良い例**: 「値が `NULL` または `''`（空文字）の場合、`CAST(0 AS INTEGER)` を設定する」
このように、**「そのままプログラムに書けるレベルの厳密さ」** で書くのがプロです。

## 3. 増分抽出の条件 (Incremental Key)
「毎日、どの行を読み込むか」を定義します。
- 更新日時 (`updated_at`) を使うのか、連番 (`id`) を使うのか。その際の「前回のしおり（しきい値）」をどこに記録するか。

---
## 🚨 The Incident: 暗黙の型変換によるデータ欠落
**事象**: ソースの `varchar(20)` をターゲットの `datetime` にマッピングしたが、特定のレコードに `2024/02/30` という不正な日付が入っており、Job全体がクラッシュした。
**対策**: マッピング表の段階で、**「変換エラー時の処理（NULLにするのか、エラー行として隔離するのか）」** を定義に含めます。
