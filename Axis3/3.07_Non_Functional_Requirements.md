# [3.07] 非機能要件（NFR）の定義：可用性と性能の「契約」

「動くのは当たり前、どう動くべきか」を合意する。

## 1. 5つの主要な非機能要件
1.  **可用性 (Availability)**: 24時間365日？ 週末のメンテナンス停止はあり？
2.  **性能 (Performance)**: バッチは何分以内に終わるべきか。
3.  **拡張性 (Scalability)**: データ量が 10倍 に増えたとき、設定変更だけで耐えられるか。
4.  **セキュリティ (Security)**: 誰が、どこから、どうやってアクセスできるか。
5.  **運用性 (Manageability)**: 障害通知はどう飛ぶか。

## 2. 実務での「要求レベル」の握り
顧客は「全部 100点」を求めますが、それは「コストが無限」になることを意味します。
- **可用性**: 99.9%（年間 8時間の停止許容）か、99.99%（同 52分）か。
- **復旧**: 障害発生から 2時間以内に復旧（RTO）させるための体制（夜間休日対応）があるか。

## 3. 目安となる「性能要件」の書き方
「速くしてほしい」は要件ではありません。
「1,000万件 の日次 ETL ジョブを、Databricks Job Cluster を使用して **30分以内** に完了させること」といった具体的な数値で定義します。

---
## 🚨 The Incident: 性能未達による検収拒否
**事象**: システムは完成したが、1ヶ月分のデータを入れたら処理に 10時間 かかってしまった。「朝の始業までに終わらないなら、このシステムはいらない」と受け取ってもらえない事態に。
**対策**: 開発の初期（プロトタイプ）段階で、本番相当のデータ量を用いた **「性能評価試験」** を実施し、非機能要件を満たせることを早期に確認します。

## 🎯 Mastery Mission: 非機能要件の策定
**タスク**: あなたは「月次決算ダッシュボード」を設計します。
- データの更新は「月に一回（月末の翌日）」だけです。
- クライアントが「絶対に止まらないように（24時間365日の死活監視）」を求めてきました。
- あなたはコスト削減のために、どのように説得しますか？（ヒント：可用性の要件とビジネス価値のバランス）
