# [3.15] インフラ基盤の結合テスト：クラウド固有の障壁を突破する

「コードは正しいのに、なぜか繋がらない」インフラ起因のバグを炙り出す。

## 1. インフラテスト (Infrastructure Testing) とは
コード（Python/SQL）のロジックではなく、Azureリソースとの疎通や設定を検証します。
- **疎通確認**: DatabricksからADLS、SQL DB、Key Vaultへ本当に接続できるか。
- **権限検証**: マネージドIDの権限が「強すぎないか」「弱すぎないか」。
- **制限検証**: VNetのスロット数不足、スロットリング（API制限）。

## 2. 実務ツール： `nc` (Netcat) と `nslookup`
DatabricksのNotebook上で以下を実行し、ネットワークの疎通を確認します。
```python
# ネットワーク疎通の確認（Pythonからシェルを実行）
%sh nc -vz my-sql-server.database.windows.net 1433
```
これで `succeeded` と出ればインフラは開通、`timed out` と出れば NSG や Firewall の設定ミスです。

## 3. 実効性の高いテスト方法
各環境（Dev/Stg/Prod）を立てた直後に、**「疎通確認専用ノートブック」** を実行します。
本格的なETLを回す前に、インフラ起因の「繋がらない」を潰しておくことで、プロジェクトの遅延を防ぎます。

## 🚨 The Incident: 本番当日、Private Link が死んでいた
**事象**: 全てのテストを終えて本番稼働させた瞬間、ADLS へのアクセスが遮断された。
**原因**: 本番用ストレージの「ファイアウォール設定」で、Databricksが属するサブネットを許可し忘れていた。
**教訓**: **「インフラの修正 ＝ 再テスト」** です。小さなネットワーク変更でも、必ずインフラテストノートブックで導通を確認しましょう。
