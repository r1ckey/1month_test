# [3.08] エラーハンドリング方針書：止める勇気と流す技術

「エラーが起きないシステムは無い。起きた時にどう振る舞うかがプロの領域。」

## 1. エラーの 3段階 分類
1.  **即時停止エラー (CRITICAL)**:
    - 認証失敗、ネットワーク遮断、ストレージ容量不足。
    - **対応**: ジョブを即座に停止し、管理者にアラートを投げる。
2.  **レコード限定エラー (ERROR)**:
    - 1行だけ型が違う、IDがNULL。
    - **対応**: その行だけ「隔離（Quarantine）」し、ジョブ全体は完遂させる。
3.  **警告 (WARN)**:
    - 閾値ギリギリ、処理時間が想定より長い。
    - **対応**: ログに残すのみ。

## 2. ログメッセージの設計ルール
「エラーが発生しました」だけのログはゴミです。以下の 4要素 を含めるよう設計書で規定します。
- **いつ (When)**: タイムスタンプ
- **どこで (Where)**: ノートブック名、関数名、項番
- **何が (What)**: エラーコード、生のエラーメッセージ
- **どうすれば (Action)**: 「DBの接続設定を確認してください」などの指示

## 3. グローバル try-except の是非
Python 開発でよくある「全てを `try-except` で囲んでしまう」のは **禁忌** です。エラーの原因が隠蔽され、デバッグが不可能になります。
- 予見できるエラー（ファイル欠落など）だけを明示的にキャッチしましょう。

## 🚨 The Incident: 無限リトライ地獄
**事象**: データの型エラーで落ちた Job に対して「自動リトライ」を設定していたが、何度リトライしてもデータ自体が壊れているため失敗し続け、数万円のクラスタ費用を無駄に溶かした。
**教訓**: 「再試行して直るもの（通信エラー等）」と「再試行しても無駄なもの（定義ミス等）」を、エラーコードや内容で切り分け、無駄なリトライを防止します。
