# [3.09] 冪等性(Idempotency) とリトライポリシー：何度実行しても同じ結果に

「二回実行したら、売上が二倍になった」という事故を論理的に防止する。

## 1. 冪等性（べきとうせい）とは
「ある操作を1回行っても、複数回行っても、最終的な結果が同じである」という性質。
- **実務重要度: SS級**。
- 夜間バッチが途中で失敗し、復旧のために「再実行（Re-run）」をした際、二重取り込みが発生しない設計を指します。

## 2. 冪等性を実現する 2つのテクニック
1.  **上書き (Overwrite/Replace)**: 
    - 特定の日付（パーティション）を、常に丸ごと入れ替える。
    - `INSERT OVERWRITE TABLE sales PARTITION (date='2024-03-01')`
2.  **マージ (Merge/Upsert)**:
    - ID が一致すれば更新、なければ挿入する。
    - Delta Lake の `MERGE INTO` 文がこれに当たります。

## 3. リトライポリシー (Retry Policy)
- **再試行回数**: 通常 3回 まで。
- **待機時間（Exponential Backoff）**: 1回目失敗後 1分、2回目失敗後 5分、と間隔を広げることで、過負荷による自滅を防ぎます。

---
## 🚨 The Incident: 1ヶ月分のデータが二重計上
**事象**: `df.write.mode("append")` で実装していたが、Jobが10回リトライに失敗し、エンジニアが手動でリトライした結果、同じデータが10回分重複して保存されていた。
**対策**: 常に `append` ではなく、**「ビジネスキーを元にした MERGE」** または **「パーティション指定の OVERWRITE」** をデフォルトの設計基準にします。
