# [3.06] I/F仕様：API vs ファイル連携の設計判断基準

「システム同士はどうやって手を繋ぐか」

## 1. インターフェース（I/F）の代表的な 2つの型
1.  **ファイル連携 (Bulk File Transfer)**:
    - **方法**: ADLS や S3 に CSV/Parquet を置いて読み書きする。
    - **メリット**: 大量データ（数千万件〜）に強い。実装が単純。
2.  **API 連携 (Rest API / JDBC)**:
    - **方法**: HTTP リクエストで JSON を取得する、またはデータベースに直接繋ぐ。
    - **メリット**: リアルタイム性が高い。部分的な更新に適している。

## 2. 実務の I/F 定義項目
仕様書には以下を必ず明記します。
- **通信プロトコル**: HTTPS, SFTP, JDBC, etc.
- **文字コード**: UTF-8 (BOMなし) と指定しないと、日本語が文字化けします。
- **改行コード**: `LF` か `CRLF` か。
- **認証方式**: OAuth2.0, API Key, Client Secret.

## 3. なぜ「ファイル連携」が無難なのか？
Databricks のような Spark 基盤では、API で 1行ずつ 100万回 叩くのは非常に非効率です。
「一度ファイルに書き出してから、一括で読み込む」プールの形をとる（ステージング）のが、データエンジニアリングの王道です。

## 🚨 The Incident: カラム位置のズレでデータ汚染
**事象**: CSV 連携で、送り側が途中に 1列 追加した。Databricks 側はカラム位置で読んでいたため、全てのデータが 1つずつ ズレて保存され、数日分のデータがゴミになった。
**対策**: I/F 設計で、**「カラムヘッダー（1行目）」** を付けることを義務付けるか、**JSON/Parquet** などのスキーマ情報を持つ形式を優先して採用します。
