# [3.05] DB論理設計：ER図とデータディクショナリの保守

「テーブル名は知っているが、そのテーブルの意味を知らない」人をゼロにする。

## 1. 論理設計 (Logical Design) とは
物理的な性能（パーティション等）を考える前に、データ同士の「関係性（リレーション）」を定義することです。

## 2. データディクショナリ (Data Dictionary)
Unity Catalog 上の `COMMENT` 機能をフル活用します。
```sql
-- テーブルにコメントを付ける（これが最強のドキュメントになる）
COMMENT ON TABLE main.silver.orders IS '販売注文の明細データ。キャンセル分も含む。';

-- カラムにコメントを付ける
ALTER TABLE main.silver.orders ALTER COLUMN order_id COMMENT '主キー。システムAから生成される12桁のID';
```
これをしておくだけで、Databricks のカタログエクスプローラ上でユーザーが自律的にデータを理解でき、あなたへの質問（ノイズ）が減ります。

## 3. ER図 (Entity Relationship Diagram)
- **マスター vs トランザクション**: 「商品マスター（1）」と「売上（多）」の関係。
- **多対多の回避**: 中間テーブル（交差テーブル）を作って、1対多の関係に分解します。

---
## 🚨 The Incident: 似たようなテーブルの乱立
**事象**: `orders`, `order_history`, `sales_data`, `old_orders` という似たようなテーブルが増え続け、どれが最新で正しいソースなのか誰もわからなくなった。
**対策**: Unity Catalog の **「リネージ（Lineage）」** 機能を使い、どのテーブルが一番信頼できるもの（Trusted source）であるかを可視化し、不要なテーブルは定期的に整理（DEPRECATED 付け）します。
