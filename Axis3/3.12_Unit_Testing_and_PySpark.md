# [3.12] 単体テスト (UT) 戦略：pytest と PySpark の組み合わせ

「バカ正直にクラスタを動かさずに、ロジックだけを秒速で検証する」

## 1. なぜ PySpark の単体テストは難しいのか
通常、PySpark を動かすには Databricks クラスタ（Spark Session）が必要です。しかし、テストのたびに 5分 かけてクラスタを立てるのは非効率です。

## 2. 実務の解：ローカル `spark-session` の活用
`pytest` と組み合わせて、ノートブック内の「ロジック関数」だけを抽出してテストします。
```python
# logic.py (テスト対象のロジック)
def calculate_tax(df):
    return df.withColumn("tax_amount", df.amount * 0.1)

# test_logic.py (テストコード)
def test_calculate_tax(spark_session):
    data = [ (1000,), (2000,) ]
    df = spark_session.createDataFrame(data, ["amount"])
    
    result_df = calculate_tax(df)
    
    assert result_df.collect()[0]["tax_amount"] == 100
```

## 3. テストファースト (TDD) の推奨
複雑な Silver 層のクレンジングロジックを書く際は、先に「期待される結果」を入力したテストコードを書き、それをパスするように本体のロジックを組むことで、バグの混入を 80% 削減できます。

## 🚨 The Incident: ノートブックが巨大すぎてテスト不能
**事象**: 1つのノートブックに 3000行 のコードが並んでおり、どこで何が起きているかわからず、一部を修正するたびに他の場所が壊れる（デグレーション）ようになった。
**対策**: ノートブック内のロジックを **「小さな関数」** に分割し、それぞれに対して単体テストをかける疎結合な設計にリファクタリングします。
